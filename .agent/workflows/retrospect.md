## /retrospect — 세션 자가 개선 회고

### 원칙

> 회고의 대상은 세 가지다: **개발 과정**, **AI 협업 과정**, **워크플로우**.
> 각 관점에서 무엇이 좋았고, 무엇이 아쉬웠는지를 KPT로 평가한다.
> Problem과 Try는 반드시 **구체적 개선 행동**으로 귀결된다.
> 각 관점의 평가가 끝나면 **자가 점검**으로 평가 자체를 점검한다.

> **분류**: 리프. 다른 워크플로우를 호출하지 않는다.

### 절차

1. **세션 요약**
   - 이번 세션의 목표와 결과를 한 줄로 정리한다.
   - 사용한 workflow들을 나열하고, 각각의 호출 맥락을 기록한다.

2. **rules 숙지**
   - `.agent/rules.md`를 읽는다. 이것이 회고의 기준선이다.
   - rules의 핵심 원칙을 머릿속에 로딩한 뒤 다음 단계로 넘어간다.

3. **KPT 평가** — 세 관점을 순서대로 수행. 각 관점 완료 후 `/review`(인라인).

   #### 🔧 개발 과정 (Development)
   > 기술적 의사결정, 설계, 디버깅, 구현 흐름을 평가한다.

   - **Keep 🟢** — 효과적이었던 기술적 판단이나 접근 방식.
   - **Problem 🔴** — 삽질, 잘못된 설계 판단, 놓친 엣지케이스 등.
   - **Try 🔵** — 다음에 같은 상황을 만나면 어떻게 할 것인지.
   - **`자가 점검`** 🪞 — 위 평가가 rules.md 원칙에 비추어 정직한가? 핵심을 놓치지 않았는가?

   #### 🤝 AI 협업 과정 (Collaboration)
   > 사용자와 AI 사이의 소통, 지시, 맥락 전달을 평가한다.

   - **Keep 🟢** — 잘 통한 지시 방식, 효과적인 맥락 공유.
   - **Problem 🔴** — 의도 오해, 불필요한 반복, 맥락 유실, 과잉/부족 행동.
   - **Try 🔵** — 소통 패턴 개선안. (예: 프롬프트 템플릿, 규칙 추가 등)
   - **`자가 점검`** 🪞 — AI 탓으로 돌린 것 중 사실은 지시가 모호했던 건 없는가? 반대로 AI의 구조적 한계를 간과하지 않았는가?

   #### ⚙️ 워크플로우 (Workflow)
   > workflow 절차 자체의 완성도를 평가한다.

   - **Keep 🟢** — 잘 작동한 단계. 계속 유지할 것.
   - **Problem 🔴** — 절차가 빠졌거나, 불필요하거나, 순서가 잘못된 것.
     - workflow를 따랐는데도 삽질 → 절차의 결함.
     - workflow를 안 따라서 삽질 → 절차의 가시성 부족.
   - **Try 🔵** — 구체적 workflow 수정안 (단계 추가/삭제/순서 변경/설명 보강).
   - **`자가 점검`** 🪞 — Try가 증상 수정인가, 근본 원인 수정인가? 과잉 절차를 추가하려는 것은 아닌가?

4. **자가 개선 실행**
   - ⚙️ Workflow Try → 즉시 `.agent/workflows/*.md` 파일에 반영한다. diff를 보여준다.
     - 절차를 **추가**하려는 Try가 있으면, 자가 점검: "이건 과잉 처리가 아닌가? 기존 단계에 병합할 수 없나?"
   - 🤝 Collaboration Try → 필요 시 `.agent/rules.md`에 규칙을 추가한다.
   - 🔧 Development Try → 필요 시 `docs/` 또는 rules에 교훈을 기록한다.
   - 📜 **Rules 발견**: KPT와 별개로, 세션 전체를 관통하는 교훈 중 `rules.md`에 기록할 만한 것이 있는지 최대한 보수적으로 검토한다. 있으면 제안하고, 없으면 넘어간다.

5. **산출물**
   - 인라인으로 세 관점의 KPT 결과를 보여주고, 수정된 파일 목록을 제시한다.
   - 프로젝트 회고 시: `docs/1-project/{name}/retrospective.md`에도 저장.
