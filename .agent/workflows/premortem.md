## /premortem — 이미 실패했다. 왜?

### 원칙

> Pre-mortem(Gary Klein)은 **발산 도구**다.
> "이 설계/제안서가 6개월 후 실패했다. 왜 실패했는가?"라고 질문한다.
> 목적은 **시야 확장**이지, 수정 지시가 아니다.
> `/reflect`(수렴)와 쌍을 이룬다: **Pre-mortem → Reflect → Submit**.

### 절차

1. **대상 확인**
   - 추가 입력이 없으면 → 직전 작업(제안서, 설계, 코드)을 대상으로 한다.
   - 추가 입력이 있으면 → 해당 주제를 대상으로 한다.

2. **실패 가정**
   - "이 결정/설계/제안이 **이미 실패했다**"고 가정한다.
   - 시간축을 설정한다: "6개월 후", "v2 개발 시", "팀이 3명으로 늘었을 때" 등.

3. **실패 원인 나열**
   - 최소 5가지 실패 시나리오를 작성한다.
   - 각 시나리오: **원인 → 징후 → 결과** 형식.
   - 카테고리별로 나눈다:
     - 🏗️ **구조적** — 아키텍처, 의존성, 확장성
     - 🧠 **인지적** — 복잡도, 학습 곡선, 유지보수 부담
     - 🔄 **진화적** — 요구사항 변경, 기술 부채 누적
     - 🤝 **협업적** — 온보딩, 문서화, 컨벤션 충돌

4. **관점 정리**
   - 각 시나리오에 대해:
     - ⚠️ **위험도**: 높음 / 보통 / 낮음
     - 🎯 **발생 확률**: 높음 / 보통 / 낮음
     - 💡 **시사점**: 이 관점이 주는 교훈 (수정 아님, 인식)

5. **산출물**
   - 인라인으로 실패 시나리오 테이블을 보여준다.
   - **수정 지시를 하지 않는다.** "이런 관점도 있다"로 끝낸다.
   - 사용자가 원하면 `/reflect`로 이어간다.

### 사용 시점

- `/project` 워크플로우에서 proposal 작성 직후 (자동)
- 중요한 설계 결정 직전 (수동)
- `/discussion` 중 사용자가 요청할 때 (수동)
