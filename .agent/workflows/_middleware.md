## Knowledge Middleware — 공통 규약

> **이 파일은 모든 워크플로우에 암묵적으로 적용된다.**
> 별도의 `@import`나 선언이 필요 없다.
> 워크플로우를 실행하는 에이전트는 이 규약을 항상 따른다.

---

### 1. 시작 시: 관련 지식 로딩

워크플로우 Step 0에서 **해당 워크플로우가 참조하는 토픽 파일**을 읽는다.

참조할 토픽은 워크플로우의 `Step 0`에 명시되어 있다. 형식:

```markdown
### Step 0: 지식 로딩
> `.agent/knowledge/[topic].md` §[섹션]을 읽는다.
```

토픽 파일이 없으면 생성하지 않는다. 없는 것은 아직 축적된 지식이 없다는 뜻이다.

---

### 2. 실행 중: 📝 Knowledge 누적

**트리거**: 워크플로우 실행 중 다음을 발견하면 **즉시** 📝에 누적한다.

| 유형 | 트리거 | 예시 |
|------|--------|------|
| **패턴 (Pattern)** | "이렇게 하니까 됐다" | "Zone autoFocus는 bind 전에 설정해야 한다" |
| **함정 (Hazard)** | "이렇게 하면 안 됐다" | "createOsPage로 앱 기능 테스트하면 거짓 GREEN" |
| **정의 (Definition)** | "이것의 경계는 여기까지다" | "Pure Projection = 렌더링 + show/hide + ARIA" |
| **원칙 (Principle)** | "항상 이래야 한다" | "컴포넌트는 상태 변경을 결정하지 않는다" |
| **선례 (Precedent)** | "이 상황에서 이렇게 판정했다" | "TabsPattern useState 모드 토글 → ⚪ 정당한 예외" |

**형식**: 간단하게 메모로 누적한다. 정제는 종료 시.

```
📝 Knowledge (이 세션에서 발견):
- K1. [한 줄 요약] (Pattern/Hazard/Definition/Principle/Precedent)
- K2. ...
```

> **주의**: AI의 추론만으로 추가하지 않는다. 사용자가 확인/동의한 것, 또는 실행 결과로 증명된 것만 Knowledge 후보다.

---

### 3. 종료 시: 📝 반영

워크플로우 마지막 Step에서, 📝에 누적된 항목이 있으면:

1. **각 Knowledge 항목의 보관 위치를 판단**한다:

   | 지식의 성격 | 보관 위치 |
   |------------|----------|
   | 프로젝트 전체 불변 원칙 | `.agent/rules.md` |
   | 특정 도메인의 표준·패턴·정의 | `.agent/knowledge/[topic].md` |
   | OS 공식 스펙 | `docs/2-area/official/[topic]/` |

2. **기존 토픽 파일에 관련 섹션이 있으면** → 해당 섹션에 추가
3. **관련 주제가 있지만 섹션이 없으면** → 새 섹션 추가
4. **맞는 파일이 없으면** → 새 토픽 파일 생성 (`kebab-case.md`)

> 📝이 비어있으면 이 단계를 스킵한다.
> 지식이 없는데 억지로 만들지 않는다.

---

### 4. 토픽 파일 표준 구조

```markdown
# [Topic Name] — [한 줄 설명]

> [이 토픽의 핵심 1문장]

---

## Config (워크플로우가 참조하는 설정)

[이 토픽에 관련된 도구, 체크리스트, 코드 템플릿 등]

## Patterns (잘 동작하는 방법)

| 패턴 | 설명 | 발견일 |
|------|------|--------|

## Hazards (함정)

| 함정 | 결과 | 대응 | 발견일 |
|------|------|------|--------|

## Precedents (판정 선례)

| 선례 | 판정 | 이유 | 날짜 |
|------|------|------|------|
```

> `Config` 섹션이 핵심이다. 워크플로우가 "§Config를 따른다"고 참조하면,
> 프로젝트별 도구/패턴이 여기서 주입된다.
> 이것이 워크플로우 범용화의 핵심 메커니즘이다.
