## /plan — 실행 전 MECE 점검표

> **전제**: 모든 행이 Clear여야 plan 완성. Complicated/Complex 행이 있으면 즉석 해소.
> **트리거**: /discussion → Clear.
> **산출물**: 전행 Clear 변환 명세표 1개 (파일로 저장).
> **원칙**: 표에 없는 것은 하지 않는다. 표에 있는 것은 전부 한다.

---

### Step 1: 대상 전수 열거

코드베이스를 조사하여 변경 대상을 빠짐없이 나열한다.
도구: `grep_search`, `view_file_outline`, `find_by_name`

### Step 2: 변환 명세표 작성

| # | 대상 | Before | After | Cynefin | 의존 | 검증 | 위험 |
|---|------|--------|-------|---------|------|------|------|
| 1 |      |        |       |         |      |      |      |

**열 규칙**:
- **대상**: 파일명:함수명 또는 파일명:타입명. 모호한 설명 금지.
- **Before**: 현재 시그니처, 위치, 동작을 코드 레벨로 기술.
- **After**: 변환 후 시그니처, 위치, 동작을 코드 레벨로 기술.
- **Cynefin**: Clear / Complicated / Complex.
- **의존**: `→#N` 형식. 없으면 `—`.
- **검증**: `tsc 0`, `+N tests`, `기존 N tests 유지` 등 구체적.
- **위험**: 파급 범위, 깨질 수 있는 소비자.

### Step 3: 비-Clear 행 즉석 해소

Complicated 또는 Complex 행이 있으면:

1. 해당 행의 **After가 왜 확정되지 않는지** 1줄로 설명한다.
2. AI의 **약식 제안**을 제시한다: "제 판단: [A안]. 이유: [근거]."
3. 사용자와 미니 discussion으로 Clear까지 해소한다.
4. 해소되면 표의 After와 Cynefin을 갱신한다.

모든 행이 Clear가 될 때까지 반복한다.

### Step 4: MECE 점검

표 완성 후 3가지 자기 점검:

```
1. CE: 모든 행을 실행하면 목표 달성? → 아니오 → 누락 행 추가
2. ME: 중복 행? → 예 → 병합
3. No-op: Before=After? → 예 → 제거
```

### Step 5: 파일 저장

변환 명세표를 파일로 저장한다.
- 프로젝트 귀속 → `docs/1-project/[name]/notes/YYYY-MMDD-HHmm-[plan]-slug.md`
- 프로젝트 미귀속 → `docs/0-inbox/YYYY-MMDD-HHmm-[plan]-slug.md`

### Step 6: 사용자 승인 + 갈래 라우팅

전행 Clear 표를 사용자에게 제시한다. 승인 전까지 실행하지 않는다.

**라우팅 결정**을 변환 명세표 파일의 `## 라우팅` 섹션에 기록한다:

```markdown
## 라우팅
승인 후 → [갈래] — [성격 한줄 설명]
```

갈래 판정:
| 논의 성격 | 갈래 | 설명 |
|----------|------|------|
| 기존 프로젝트에 태스크 | → `/go` (기존 프로젝트) | 프로젝트명 명시 |
| 새 프로젝트 필요 | → `/project` | 스코프와 규모 명시 |
| 긴급 수정 | → `/issue` | 이슈 제목 명시 |
| 유저 스토리 | → `/stories` | 스토리 대상 명시 |
| 참고 자료 | → `/resource` | 주제 명시 |
| 아이디어 보관 | → `/backlog` | 근거 명시 |

> **`→ /go`만 쓰지 않는다.** 성격(프로젝트명, 이슈 제목 등)을 반드시 포함한다.

수정 요청:
- "N번 빠졌어" → Step 2로
- "N번 필요 없어" → 제거 후 재확인
