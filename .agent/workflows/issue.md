## /issue — 8D 기반 이슈 해결 파이프라인

> **분류**: 오케스트레이터. `/go` 보편 사이클의 **Medium 프리셋**을 따른다.
> **이론적 기반**: 8D Problem Solving (Ford Motor Company) + Cynefin Framework

### `/go` 보편 사이클과의 관계

`/issue`는 `/go`의 Medium 프리셋이다. Step 2~4(discussion, prd, redteam)를 skip하고 Step 5(/tdd)부터 시작한다.
다만 진입 전에 이슈 등록, triage, 베이스라인, 분석이 선행된다.

### 이슈 진입 절차 (보편 사이클 진입 전)

#### 1. 이슈 등록
- 위치: `docs/1-project/0-issue/YYYY-MM-DD_<슬러그>.md`
- 형식: 원문 + 환경 + 재현 단계 + 기대 결과 + 실제 결과 + 관련 이슈

#### 2. Triage (심각도 분류)
- `P0` — 빌드 불가 / 전체 라우트 영향
- `P1` — 특정 기능 완전 동작 불가
- `P2` — 부분적 이상, 대안 존재
- `P3` — cosmetic / 개선사항

#### 3. 베이스라인 확보
- `/ready` 실행 — 환경이 정상인 상태에서만 이슈 분석 가능.

#### 4. 분석 — `/diagnose` 실행
- 코드를 수정하지 않고 원인을 분석한다.
- **관련 프로젝트 문서를 먼저 읽는다** — PRD, Proposal, Status.
- Before 스냅샷 기록.

#### Gate: Cynefin 도메인 분류 + 사용자 확인
```markdown
**Clear/Complicated**: [코드에서 확인한 사실]
**Complex**: [판단이 필요한 질문]
**내 이해 1문장**: "~이렇게 동작해야 하는 것으로 이해했다"
```
- 사용자 확인 없이 추측으로 결정하면 게이트 위반.

#### Gate: 설계 냄새 검사 (오컴의 면도날)

수정안을 떠올렸으면, **코드를 고치기 전에** 다음 4개 질문을 확인한다:

| # | 질문 | "예"이면 |
|---|------|---------|
| 1 | 수정이 개체(concept)를 **늘리는가**? (플래그, 모드, 프로토콜 추가) | 설계 의심 ↓ |
| 2 | 호출자가 **내부 로직을 알아야** 하는가? (순서 의존, 숨겨진 규약) | 설계 의심 ↓ |
| 3 | 같은 종류의 버그가 **다른 호출 지점에서 또 날 수 있는가**? | 설계 의심 ↓ |
| 4 | 수정 후 **API surface가 커지는가**? | 설계 의심 ↓ |

- **모두 "아니오"** → 코드 패치 진행 (Step 5)
- **하나라도 "예"** → 코드를 고치는 것이 아니라 **API를 재설계**한다. 수정안이 아니라 대안 설계를 먼저 제시하고 사용자 확인 후 진행.

> 사례: `copyTodo({id}) × N + _multi` → 4개 질문 모두 "예" → `copyTodo({ids})` 배치 재설계.

### 보편 사이클 진입

Gate 통과 후, `/go` Medium 프리셋으로 **Step 5(/tdd)부터** 진입:
- Step 5: /tdd — 재현 테스트 (실패하는 테스트 = 이슈 존재 증명)
- Step 6: /solve — 테스트를 통과시키기 위한 수정
- Step 7~13: /review → /fix → /doubt → /cleanup → /verify → /changelog → /ready

### 이슈 종료 (보편 사이클 이후)

1. 이슈 문서에 `[Closed]` + 해결 요약 추가.
2. 이슈 파일을 `docs/1-project/0-issue/closed/`로 이동.
3. 관련 프로젝트 BOARD.md Done에 기록.
4. Step 14~16: /retrospect → /coverage(선택) → skip(/para).

### 재발 방지

근본 원인이 구조적/패턴적 문제인 경우 → `/rules` 실행하여 규칙 추가.

### 재현 실패 시 Fallback

1. Vite 터미널 로그 확인.
2. 사용자에게 콘솔 에러 요청.
3. **추측으로 원인을 단정하지 않는다.**
4. 에러 메시지를 확보한 후에만 분석 진행.
