## /backlog — 백로그 항목 등록

> 아이디어·OS gap·기술부채·개선안을 `docs/5-backlog/`에 정형화하여 등록한다.
> 다른 세션/에이전트가 독립적으로 실행할 수 있는 수준의 상세도를 보장한다.

### 원칙

1. **Self-contained**: 항목을 읽는 에이전트가 이 대화의 맥락 없이도 작업을 시작할 수 있어야 한다.
2. **Actionable**: "~하면 좋겠다"가 아니라 "무엇을 왜 어떻게"가 있어야 한다.
3. **No duplication**: 등록 전에 `docs/5-backlog/` 기존 항목과 `docs/5-backlog/os-gaps.md`를 확인한다.

### 입력

사용자가 직접 설명하거나, 대화 맥락에서 추출한다.

### 항목 분류

| 태그 | 설명 | 예시 |
|------|------|------|
| `os-gap` | OS가 아직 제공하지 않는 프리미티브 | OG-XXX → `os-gaps.md`에 등록 |
| `tech-debt` | 작동하지만 개선이 필요한 코드/구조 | 레거시 패턴, 중복 코드 |
| `idea` | 새 기능·패턴·도구 아이디어 | 새 워크플로우, 새 앱 |
| `infra` | 개발 인프라·DX 개선 | 테스트 도구, CI, 문서화 |
| `analysis` | 조사·분석이 필요한 주제 | 아키텍처 탐색, 패턴 비교 |

### 실행

#### Step 1: 중복 확인

```bash
ls docs/5-backlog/
```

기존 항목에 이미 같은 주제가 있으면 → 기존 파일에 추가/갱신한다 (새 파일 생성 안 함).

#### Step 2: 파일 생성

- **OS gap**: `docs/5-backlog/os-gaps.md` 테이블에 행 추가 (OG-XXX 번호 부여)
- **그 외**: `docs/5-backlog/{slug}.md` 파일 생성

파일명 규칙:
- kebab-case
- 날짜 접두사 선택적: `YYYY-MMDD-` (시간 맥락이 중요할 때만)
- `[{태그}]` 접두사 선택적: `[analysis]-topic.md`

#### Step 3: 항목 작성

```markdown
# {제목}

> 작성일: YYYY-MM-DD
> 태그: {os-gap | tech-debt | idea | infra | analysis}
> 우선순위: {P0 | P1 | P2 | P3} (선택)

## 문제 / 동기

무엇이 문제인가? 왜 이것이 필요한가?
- 구체적 증상 또는 필요성
- 발견 경위 (어떤 작업 중 발견했는가)

## 현재 상태

지금은 어떻게 되어 있는가?
- 관련 파일/모듈 경로
- 현재 동작 방식

## 기대 상태

어떻게 되어야 하는가?
- 완료 조건 (Done Criteria)
- 검증 방법

## 접근 방향 (선택)

어떻게 해결할 수 있는가?
- 후보 접근법
- 참고 자료/선례

## 관련 항목

- 연관된 다른 백로그/프로젝트/이슈 링크
```

#### Step 4: 확인

작성한 항목을 사용자에게 보여주고 확인받는다.

### 주의

- **백로그는 실행이 아니다.** 항목을 등록만 하고, 실행은 `/go`나 `/project`가 담당한다.
- **os-gaps.md는 특수 파일이다.** OS gap은 반드시 이 파일의 테이블에 등록한다. 별도 파일을 만들지 않는다.
- 대화 중 `/backlog`를 호출하면 현재 논의를 중단하지 않고, 항목만 등록한 뒤 원래 흐름으로 복귀한다.
