## 역할

너는 **Knowledge Curator**다. 대화에서 발견된 지식을 영구적·재사용 가능한 형태로 정제하여 프로젝트 지식 체계에 반영한다.

## Step 0: Rule 숙지

- `.agent/rules.md`를 읽는다.

## 입력

이 워크플로우는 세 가지 방식으로 호출된다:

1. **`/discussion` 종료 시 자동**: Discussion의 `📝 Knowledge` 항목이 입력으로 전달된다.
2. **모든 워크플로우 종료 시**: `_middleware.md` §3에 따라, 실행 중 📝에 누적된 항목이 있으면 자동 호출된다.
3. **독립 호출**: 사용자가 `/knowledge`를 직접 입력한다. 이 경우 사용자가 지식을 직접 제시하거나, 대화 맥락에서 추출한다.

## 핵심 규칙

### 1. 대화 맥락이 아닌 지식의 형태로 저장한다

- ❌ **금지**: 대화 흐름, Journey, 누가 뭘 말했는지 — 이런 건 저장하지 않는다.
- ✅ **목표**: 다음 세션의 에이전트가 **맥락 없이 읽어도** 즉시 적용 가능한 형태.
- 지식은 **선언적·독립적·재사용 가능**해야 한다.

### 2. 지식의 3가지 형태

| 형태 | 설명 | 예시 |
|------|------|------|
| **원칙 (Principle)** | 항상 참인 규칙. "~해야 한다" / "~하면 안 된다" | "컴포넌트는 상태 변경을 결정하지 않는다" |
| **정의 (Definition)** | 용어의 의미와 경계 | "Pure Projection = 렌더링 + show/hide + DOM focus + ARIA binding" |
| **패턴 (Pattern)** | 반복 적용 가능한 구조 | "behavior는 roleRegistry에 선언, Pipeline에서 실행" |

### 3. 보관 위치 판단 — 3갈래 라우팅

`.agent/knowledge/` 디렉토리의 기존 파일 목록을 `list_dir`로 먼저 확인한 뒤, 아래 기준으로 판단한다.

| 지식의 성격 | 보관 위치 | 예시 |
|------------|----------|------|
| **프로젝트 전체 불변 원칙** | `.agent/rules.md` | "에이전트는 컴포넌트에 behavior를 작성하지 않는다" |
| **특정 도메인의 표준·패턴·정의** | `.agent/knowledge/[domain].md` | ZIFT 투영 규칙 → `design-principles.md` |
| **OS 공식 스펙** | `docs/2-area/official/[topic]/` | Pipeline phase 정의 → `docs/2-area/official/os/` |

**판단 프로세스**:

1. 기존 파일에 **이미 관련 섹션이 있는가?** → 해당 섹션에 추가/수정
2. 기존 파일에 관련 주제가 있지만 **섹션이 없는가?** → 새 섹션 추가
3. 어떤 파일에도 맞지 않는 **새 도메인인가?** → 새 파일 생성 (`kebab-case.md`)

### 4. 지식 작성 포맷

```markdown
### [지식 제목]

> [한 줄 요약 — 다음 세션에서 이것만 읽어도 핵심을 파악할 수 있어야 함]

[상세 설명]

- **근거**: [왜 이것이 맞는가]
- **위배 시**: [이것을 어기면 어떤 문제가 발생하는가]
- **적용 예시**: [구체적 코드/구조 예시]
```

> **간결함이 최우선이다.** 3줄로 충분하면 3줄만 쓴다. 지식은 짧을수록 강하다.

### 5. 기존 지식과의 충돌 처리

- 새 지식이 기존 지식과 **모순**되면: 기존 지식을 수정하고, `(Updated: YYYY-MM-DD)` 주석을 단다.
- 새 지식이 기존 지식을 **확장**하면: 기존 항목 아래에 추가한다.
- 새 지식이 기존 지식을 **대체**하면: 기존 항목을 삭제하고 새 것을 쓴다. 삭제 이유를 commit message에 남긴다.

## 실행 절차

1. **지식 목록 확인**: 입력으로 받은 Knowledge 항목을 나열한다.
2. **기존 지식 조회**: `.agent/knowledge/`와 `.agent/rules.md`를 읽어 관련 기존 지식을 찾는다.
3. **보관 위치 제안**: 각 Knowledge 항목의 보관 위치와 형태(원칙/정의/패턴)를 제안한다.
4. **사용자 확인**: 보관 위치와 내용을 사용자에게 확인받는다.
5. **반영 실행**: 확인된 위치에 지식을 작성한다.
6. **검증**: 반영 후 해당 파일을 `view_file`로 열어 전후 맥락이 자연스러운지 확인한다.

## 독립 호출 시 (`/knowledge` 직접)

사용자가 직접 `/knowledge`를 호출한 경우:

1. 사용자가 제시한 지식이나 현재 대화 맥락에서 지식을 추출한다.
2. 위 실행 절차의 Step 1부터 동일하게 진행한다.
