# Interactive OS 선언문

## Goal — 우리가 이루려는 것

1. **핸들러 지옥을 끝낸다.** onClick, onChange, useEffect에 흩어진 로직을 하나의 파이프라인으로 통합한다.
2. **웹 앱에 OS의 질서를 부여한다.** 포커스, 네비게이션, 접근성 — 개별 앱이 각자 구현하는 게 아니라, 시스템이 보장한다.
3. **AI와 인간이 같은 도구로 개발한다.** AI가 읽고, 쓰고, 테스트하고, 디버깅하는 모든 과정이 인간과 동일한 인터페이스 위에 있다.
4. **접근성은 기능이 아니라 인프라다.** ARIA, 키보드 네비게이션은 "나중에 추가"하는 게 아니라 시스템의 기본 동작이다.
5. **복잡한 인터랙션을 단순한 프리미티브로 조립한다.** Zone, Item, Trigger — 작은 블록의 조합으로 어떤 인터랙션이든 표현할 수 있다.
6. **모든 상태 변화를 관찰하고 되돌릴 수 있다.** Transaction 로그, 타임 트래블, Inspector — 시스템은 항상 투명하다.
7. **구조가 문서를 대체한다.** 코드 구조 자체가 스펙이다. 구조를 보면 설계가 읽힌다.
8. **AI가 강해질수록 시스템도 강해진다.** AI에게 유리한 구조(순수함수, 브랜드 타입, 단일 파이프라인)가 인간에게도 유리하다.
9. **하나의 앱이 아니라 플랫폼을 만든다.** Todo, Builder는 시작일 뿐이다. 어떤 앱이든 이 OS 위에서 동작할 수 있다.
10. **이 시스템 자체가 증명이다.** 이론이 아니라 동작하는 코드로 비전을 증명한다.

## Project — 우리가 믿는 것

1. **앱은 페이지의 모음이 아니라, 운영체제다.** 사용자의 모든 의도는 커맨드로 표현되고, 커맨드는 시스템 어디서든 동일하게 동작한다.
2. **번역기는 번역만 한다.** 입력을 커맨드로 바꾸는 자와, 커맨드를 실행하는 자는 서로를 모른다.
3. **모든 변경은 하나의 문을 통과한다.** 상태 변경의 경로가 둘이면, 버그의 경로도 둘이다. 단일 파이프라인.
4. **행동은 선언으로 결정된다.** 같은 커맨드가 맥락에 따라 다르게 동작한다. 코드가 아니라 설정이 행동을 결정한다.
5. **모든 코드는 부채다.** 존재하는 코드는 정당화되어야 한다. 미래를 위한 추측은 현재의 복잡성이다.
6. **이름은 법이다.** 하나의 개념에 하나의 이름. grep 한 번이면 모든 연결이 보여야 한다.
7. **증명 없는 통과는 통과가 아니다.** 빌드가 됐다는 건 안 깨졌다는 증거가 아니다. 런타임에서 증명하라.
8. **표준이 있으면 발명하지 않는다.** W3C가 정의한 것은 우리의 원칙이다. 검증된 해법이 커스텀 구현에 앞선다.
9. **AI가 실수해도 구조가 잡아준다.** 관찰 가능하고, 검증 가능하고, 재현 가능하고, 복구 가능한 시스템.
10. **불확실하면 나누고, 나눠도 모르면 묻는다.** 정답이 보이면 증명하고 실행한다. 안 보이면 더 작게 나눈다. 그래도 안 보이면 혼자 끙끙대지 말고 묻는다.

## Working — 우리가 일하는 방식

1. **코드를 고치기 전에 정답인지 판단한다.** 수정 중에 판단하면 이미 늦다. 정답이 보이면 증명하고 실행. 안 보이면 나누거나 묻는다. → `/divide`
2. **가장 빠른 피드백부터 잡는다.** 순수함수 테스트 → 커맨드 테스트 → E2E → 수동 확인. 느린 루프부터 시작하지 않는다. → `/fix`
3. **아래 레이어가 검증되어야 위를 신뢰할 수 있다.** E2E가 실패하면 E2E를 고치는 게 아니라, 가장 아래 레이어부터 격리해서 올라간다. → `/diagnose`
4. **분석은 코드 수정보다 가치 있다.** "뭘 고쳤다"보다 "왜 깨졌는지 알아냈다"가 더 큰 산출물이다. → `/diagnose`, `/inbox`
5. **작게 커밋하고, 자주 증명한다.** 거대한 한 방이 아니라, 증명된 작은 단계의 연속으로 전진한다. → `/cleanup`
6. **수동 확인은 증거가 아니다.** 눈으로 봤다는 건 다음에 또 눈으로 봐야 한다는 뜻이다. 자동화된 검증만 증거다. → `/test`
7. **다 하려고 하지 않는다.** 스코프를 좁히고, 하나를 끝내고, 다음으로 넘어간다. 세 개를 동시에 반쯤 하느니 하나를 완전히 끝낸다. → `/divide`
8. **모르면 묻는다. 추측으로 구현하지 않는다.** 잘못된 방향으로 100줄 쓰는 것보다 질문 하나가 낫다. → `/discussion`
9. **도구가 있으면 도구를 쓴다.** 워크플로, TestBot, Inspector — 이미 만들어둔 것을 무시하고 맨손으로 하지 않는다.
10. **깨진 것을 보면 지나치지 않는다.** 내 작업 범위가 아니어도, 발견한 문제는 기록한다. 지금 안 고쳐도 잊히지 않게 한다. → `/inbox`
11. **브라우저는 최후의 보루다.** 검증은 테스트를 작성해서 한다. 순수함수 → 커맨드 → E2E(TestBot/Playwright) 순으로 테스트를 짜고, 테스트로 검증할 수 없는 경우에만 브라우저 에이전트를 꺼낸다. → `/test`, `/fix`
