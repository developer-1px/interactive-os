# Discussion Archiving — Conclusion

## Why

디스커션 로그는 "제일 좋은" 사고 기록이지만, 현재 flat + 2파일/세션 구조가 스케일하지 않는다. 34개 파일이 6일간 쌓였고, 네이밍 불일치로 탐색이 어렵다. 디스커션의 가치를 보존하면서 탐색 가능하고 스케일러블한 구조가 필요하다.

## Intent

디스커션을 "프로젝트 씨앗"과 "독립 에세이(블로그)"로 이원화하고, 1파일 통합 + 주제 slug 필수 + 프로젝트 승격 시 이동(move) 구조를 만들어, 자가 정리(self-pruning)되는 아카이브를 구축한다.

## Warrants (논거)

**W1. 디스커션은 가장 가치 있는 사고 기록이다**
- 사용자: "제일 좋고 디스커션 로그가 좋은데"
- 나중에는 추억, 다른 사람에게는 why와 history

**W2. 현재 flat + 2파일/세션 구조는 스케일하지 않는다**
- 34개 파일이 한 폴더에 flat, 6일간 평균 5~6개/일 생성
- "너무 중구난방으로 커질 것 같음"

**W3. 네이밍 불일치가 탐색을 더 어렵게 만든다**
- 초기: `2026-02-07-stream-inspector-discussion.md` (주제 있음)
- 최근: `2026-0213-0824-discussion-conclusion.md` (주제 없음)
- 워크플로우 템플릿이 주제 slug를 누락

**W4. Journey가 Conclusion보다 더 가치 있다 — 과정 자체가 핵심**
- "journey가 더 가치있어. 기록 자체가 가치야"

**W5. 접근 패턴은 "가끔 열람"이지 "자주 검색"이 아니다**
- "보관과 열람용인데 자주 하지는 않을거야"

**W6. 프로젝트 승격 시 이동(복사 아님)으로 중복을 제거한다**
- "사본이어야 할 필요가 있나? 프로젝트로 승격이이나 이동을 시켜버리자"
- `/project` 워크플로우의 Step 3에서 관련 문서 이동

**W7. 프로젝트화되지 않은 디스커션 = 독립 에세이 = 블로그 성격**
- "나머지는 blog에 가까운데"
- 프로젝트로 간 것은 빠져나가고, 남는 건 독립적 사유

**W8. 파일명에 topic-slug 필수 — 당연한 건데 워크플로우 템플릿이 누락했었음**
- "세상에 지금 파일명에 주제를 안 남겨?"
- 형식: `YYYY-MMDD-HHmm-{topic-slug}-{type}.md`

**W9. Journey→Conclusion 순서로 1파일 통합 — 독자의 인지 흐름에 맞춤**
- "대화를 읽고 정리된걸 봐야 보일테니까"
- 하나의 글로서 완결성: 내러티브(journey) + 정리(conclusion)

**W10. 프로젝트 폴더 구조 간소화**
- Before: `0-discussion-journey.md` + `1-discussion-conclusion.md`
- After: `0-discussion.md` (1파일)
- 번호가 하나씩 앞당겨지고 더 깔끔해짐

**W11. 생성 과정은 2파일 분리 → 합치기 — 산출물 형태 ≠ 생성 과정**
- "파일을 생성할때는 2개를 만들고 합치는거야. 한번에 만들면 퀄리티가 별로더라"
- conclusion 작성(논증 구조에 집중) → journey 작성(서사 흐름에 집중) → 통합

**W12. 폴더명 `11-discussions` 유지**
- "응 일단 유지해 넘버링은 나중에 바꾸더라도 discussions는 필요해"

**W13. 소급 정리 범위 확정**
- "프로젝트로 넘어간건 제거하고 소급정리"
- 34개 파일을 1파일 통합 + topic-slug 추가

## 결론 — 디스커션 아카이빙 새 규칙

### 파일 구조
```
생성 과정:
  1. conclusion.md 작성 (논증 구조 집중)
  2. journey.md 작성 (서사 흐름 집중)
  3. 통합: journey(上) + conclusion(下) → 1파일
  4. 임시 2파일 삭제

최종 산출물:
  파일명: YYYY-MMDD-HHmm-{topic-slug}.md
  위치: docs/11-discussions/
  내용: Journey → Conclusion 순서
```

### Lifecycle
```
디스커션 발생 → 11-discussions/에 저장 (1파일)
                    ↓
        ┌─── 프로젝트 승격? ───┐
        │ YES                   │ NO
        ↓                      ↓
  1-project/[name]/        11-discussions/에 잔류
  0-discussion.md (이동)     (= 블로그 포스트)
        ↓
  프로젝트 종료 →
  4-archive/YYYY/[name]/
```

### 자가 정리 메커니즘
- 프로젝트 승격 시 **이동(move)** → 원본 제거 → 중복 없음
- `11-discussions/`는 자연스럽게 **self-pruning**
- 남는 것은 오직 "독립적 사유" = 블로그

### 소급 정리
- 기존 34개 파일을 쌍으로 병합
- topic-slug 추가 (파일명 또는 내용에서 추출)
- 프로젝트로 이미 승격된 것 확인 및 원본 제거

## 한 줄 요약

디스커션을 1파일(journey→conclusion) + topic-slug 필수 + 프로젝트 이동(move) 구조로 재설계하여, 프로젝트 씨앗과 독립 에세이로 이원화하고 자가 정리되는 아카이브를 구축한다.
