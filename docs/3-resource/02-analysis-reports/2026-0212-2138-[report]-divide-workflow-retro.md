# /divide 워크플로우 첫 사용 회고

| 항목 | 내용 |
|------|------|
| 원문 | divide 해보니 어때? 회고해 |
| 내가 추정한 의도 | /divide를 실전에 써본 AI의 행동 패턴을 점검하고, 워크플로우 자체를 개선할 단서를 찾으려는 것 |
| 날짜 | 2026-02-12 |
| 적용 대상 | Todo E2E 12개 테스트 실패 디버깅 |

## 1. 개요

`/divide`를 Todo E2E 테스트 실패(12개 중 5~6개 fail)에 적용했다. "정답 있는가?"를 기준으로 분해하고, 정답 있는 것은 바로 실행, 없는 것은 사용자에게 질문하라는 원칙이었다.

## 2. 실제로 어떻게 행동했는가

### 잘 된 것

- **정답 있는 5개 버그를 연쇄적으로 수정했다** — `isContentEditable` false, registry lookup 불일치, draft 미지원, DOM 미정리, payload 무시. 한 레이어 고치니 다음 레이어 문제가 드러나는 구조.
- **"정답 없음" 판단 후 멈췄다** — Meta+Arrow 리매핑 충돌은 설계 결정이 필요하다고 판단하고 사용자에게 선택지를 제시.
- **보고서를 산출물로 남겼다** — 뭘 고쳤고, 뭘 못 고쳤는지 기록.

### 잘 안 된 것

| 문제 | 구체적 사례 | 원인 |
|------|------------|------|
| **"정답 있음"인데 빙빙 돌았다** | FIELD_COMMIT에서 field lookup을 3번 고침 (getActiveField → iterate → mode 기반) | 한 번에 정확한 원인을 못 찾고 시행착오. 코드를 읽는 대신 고치고 돌려보는 패턴 |
| **브라우저 디버깅 남용** | E2E가 있는데 browser_subagent로 수동 클릭 반복 (사용자가 지적) | "눈으로 확인"하려는 습관. E2E + console.log가 10배 빠름 |
| **나누기 전에 이미 고치고 있었다** | 최초에 레이어 분해 없이 바로 FIELD_COMMIT 코드를 수정 시작 | divide 원칙을 선언만 하고, 실제로 "먼저 나누기"를 건너뜀 |
| **"정답 없음"을 너무 늦게 판단** | Meta+Arrow 문제를 getCanonicalKey 코드를 읽고 나서야 "설계 결정"이라고 인식 | 키바인딩 테이블만 봤어도 바로 알 수 있었음 |

## 3. /divide 워크플로우 개선 제안

### 현재 버전의 약점

1. **"나눠"가 너무 추상적** — 뭘 기준으로 나눌지 가이드가 없다. 실전에서는 그냥 코드를 고치기 시작함.
2. **"정답 있는가?" 판단 시점이 명시 안 됨** — 코드 수정 도중에 판단하면 이미 늦다. 수정 전에 판단해야 한다.
3. **디버깅 전략이 없다** — "어떻게 확인할 것인가"에 대한 가이드가 없어서 브라우저 디버깅 같은 비효율적 방법을 택했다.

### 추가하면 좋을 단계

```diff
 1. 대상 분석
+1.5 **증거 수집 전략 결정** — E2E 로그? console.log? 단위 테스트? 가장 빠른 피드백 루프를 선택
 2. 판단: 정답이 있는가?
+   - 코드를 수정하기 전에 판단한다
+   - "읽기만으로 판단 가능"이면 Known
+   - "고쳐봐야 아는" 것도 Known (시행착오일 뿐)
+   - "어떻게 고쳐야 할지 선택지가 여럿"이면 Open
 3. 분해
 4. 모르겠으면 질문
 5. 보고서
```

## 4. 해법 유형

🟡 **Constrained** — /divide 원칙 자체는 유효하지만, AI가 실제로 따르게 만들려면 더 구체적인 체크포인트(증거 수집 전략, 판단 시점)가 필요하다.

## 5. 인식 한계

- 이 회고는 1회 적용 사례에 기반한다. 다른 유형의 문제(설계, 리팩토링)에 적용하면 다른 패턴이 나올 수 있다.
- "잘 안 된 것"은 AI 자신의 행동 분석이므로, 사용자가 관찰한 것과 다를 수 있다.

## 6. 열린 질문

1. /divide 워크플로우에 "증거 수집 전략" 단계를 추가할까?
2. "정답 판단"의 체크리스트를 더 구체화할까?
3. 이 회고 자체를 /divide 워크플로우 파일에 반영할까?

---

**한줄요약**: /divide는 "정답 있는 5개 버그"를 연쇄 수정하는 데 효과적이었으나, AI가 "먼저 나누기"를 건너뛰고 바로 코드 수정에 들어가는 경향이 있어 증거 수집 전략과 판단 시점을 명시하는 개선이 필요하다.
