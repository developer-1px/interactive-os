## 이전 편 요약

3편에서 발견한 것: 파이프라인은 직선이 아닙니다. happy path + 복구 경로 + 브레이크 포인트가 필요합니다. 그리고 같은 문제가 스케일을 바꿔 반복됩니다 (프랙탈).

이번 편에서는 마지막 퍼즐 — LLM이 **같은 실수를 반복하지 않게** 하는 방법, 그리고 현재까지 도달한 곳을 다룹니다.

## 1. "성장을 못한다" — 맨 처음 메모로 돌아가기

이 시리즈의 맨 처음 메모에 이런 말이 있었습니다:

> "매번 새로운 신입이 오는데 오자마자 일하고 싶다."
> "성장을 못한다. 학습을 못한다."

LLM은 세션마다 컨텍스트가 리셋됩니다. 지난 세션에서 "이렇게 하면 안 된다"를 배워도, 다음 세션에서는 처음부터 시작합니다. 사람의 팀에서 매일 신입이 교체되는 것과 같습니다.

파이프라인은 "뭘 해야 하는가"를 구조화했습니다. 하지만 **"뭘 하면 안 되는가"**는 여전히 매 세션 반복 학습해야 했습니다. 워크플로우 규칙에 "이거 하지 마"를 써놔도, 결국 같은 실수가 터집니다. 왜일까요? 워크플로우는 50줄인데, 거기에 프로젝트 전체의 함정과 패턴을 다 넣을 수는 없기 때문입니다.

## 2. 외부 두뇌 — Knowledge Middleware

이 문제의 해법은 **기억을 LLM 밖에 저장하는 것**입니다.

J.C.R. Licklider는 1960년에 "Man-Computer Symbiosis"라는 논문에서, 인간의 인지 능력을 외부 시스템으로 확장하는 개념을 제안했습니다. 이후 이 아이디어는 **Exocortex**(외부 두뇌)라는 이름으로 발전했습니다. LLM에게도 같은 원리가 적용됩니다:

> **LLM 자체는 학습 못 하지만, 학습 결과를 외부에 저장하면 다음 세션이 "학습한 척" 할 수 있습니다.**

| | 사람 | LLM (기본) | LLM (파이프라인) |
|---|------|-----------|----------------|
| 실수 후 | 기억 → 안 반복 | 리셋 → 반복 | **기록 → 다음 세션이 읽음 → 안 반복** |

이 프로젝트에서 이걸 구현한 것이 `_middleware.md` — **Knowledge Middleware**입니다. 이름 그대로, 모든 워크플로우에 암묵적으로 적용되는 미들웨어입니다.

```markdown
## Knowledge Middleware — 공통 규약

> **이 파일은 모든 워크플로우에 암묵적으로 적용된다.**
> 별도의 @import나 선언이 필요 없다.
> 워크플로우를 실행하는 에이전트는 이 규약을 항상 따른다.
```

동작은 세 단계입니다:

### 시작 시: 지식 로딩

```markdown
### 1. 시작 시: 관련 지식 로딩

워크플로우 Step 0에서 해당 워크플로우가 참조하는 토픽 파일을 읽는다.
```

`/red`를 실행하면 `testing-hazards.md`와 `testing-tools.md`를 먼저 읽습니다. 여기에 이전 세션들이 축적한 함정과 패턴이 있습니다. 새 세션은 이걸 읽고 시작하니까, 이전 세션이 겪은 실수를 반복하지 않습니다.

### 실행 중: 발견 누적

```markdown
### 2. 실행 중: 📝 Knowledge 누적

| 유형 | 트리거 |
|------|--------|
| 패턴 (Pattern) | "이렇게 하니까 됐다" |
| 함정 (Hazard)  | "이렇게 하면 안 됐다" |
| 정의 (Definition) | "이것의 경계는 여기까지다" |
| 원칙 (Principle) | "항상 이래야 한다" |
| 선례 (Precedent) | "이 상황에서 이렇게 판정했다" |
```

워크플로우 실행 중에 새로운 발견이 있으면, 유형별로 태깅하여 누적합니다. 중요한 제약이 있습니다: **AI의 추론만으로 추가하지 않습니다. 사용자가 확인/동의한 것, 또는 실행 결과로 증명된 것만 Knowledge 후보입니다.**

### 종료 시: 영구 반영

```markdown
### 3. 종료 시: 📝 반영

| 지식의 성격 | 보관 위치 |
|------------|----------|
| 프로젝트 전체 불변 원칙 | .agent/rules.md |
| 특정 도메인의 표준·패턴·정의 | .agent/knowledge/[topic].md |
| OS 공식 스펙 | docs/2-area/official/[topic]/ |
```

세션이 끝나면, 누적된 Knowledge를 적절한 파일에 영구 반영합니다. 다음 세션이 시작될 때 이 파일이 로딩되어, **이전 세션의 학습이 이어집니다.**

이 구조가 만드는 순환입니다:

```
세션 N:  지식 로딩 → 워크플로우 실행 → 새 발견 → 지식 저장
                                                    ↓
세션 N+1: 지식 로딩 → 워크플로우 실행 → 새 발견 → 지식 저장
                                                    ↓
세션 N+2: ...
```

지식 파일의 실물을 보면, 이 순환이 실제로 작동한 흔적이 보입니다:

```markdown
# testing-hazards.md (실제 파일에서 발췌)

## Hazards (함정)

| 함정 | 결과 | 대응 |
|------|------|------|
| createOsPage로 앱 기능 테스트 | 거짓 GREEN | createTestPage 사용 |
| os.dispatch() 직접 호출 테스트 | 파이프라인 미검증 | Input-First Testing |
```

이 함정들은 실제 세션에서 겪은 것이 기록된 것입니다. 다음 세션의 `/red`가 이 파일을 읽으면, 같은 함정에 빠지지 않습니다.

### 지식 인프라 — 컨테이너도 필요합니다

Knowledge Middleware가 지식의 **내용**을 다룬다면, 지식의 **컨테이너**(어디에 넣고, 어디서 꺼내고, 언제 버리는가)도 필요합니다. 이 프로젝트에서는 PARA 시스템을 기반으로 세 개의 워크플로우가 이 역할을 합니다:

| 워크플로우 | 역할 | 실전 증거 |
|-----------|------|----------|
| `/inbox` | 입구 — 요청을 정형화하여 `docs/0-inbox/`에 저장 | `b68a59d8` inbox workflow 단순화 |
| `/status` | 현황판 — STATUS.md를 자동 갱신하여 프로젝트 대시보드 유지 | Information Radiator의 실제 구현체 |
| `/retire` | 출구 — superseded된 문서를 AI 컨텍스트에서 제거 | `70eea9c0` 13개 프로젝트 일괄 retire |

특히 `/retire`가 중요합니다. 지식은 쌓이기만 하면 오히려 해가 됩니다 — 오래된 정보가 새 세션의 컨텍스트를 오염시킵니다. 2026년 3월 3일, 13개 완료 프로젝트를 일괄 retire하고 510개 파일을 deep archive한 것(`85b0c6c0`)은 **지식을 죽이는 것도 학습의 일부**라는 증거입니다.

명시적 지식 관리를 위한 `/knowledge` 워크플로우도 있습니다. `_middleware`가 암묵적(모든 워크플로우에 자동 적용)이라면, `/knowledge`는 명시적으로 "이 발견을 어디에 저장할지" 판단하고 생성하는 전용 워크플로우입니다.

## 3. `/retrospect` — 구조적 회고

Knowledge Middleware가 **실행 중** 학습이라면, `/retrospect`는 **세션 종료 후** 학습입니다.

```markdown
## /retrospect — 세션 자가 개선 회고

> 회고의 대상은 세 가지다: 개발 과정, AI 협업 과정, 워크플로우.
> 각 관점에서 무엇이 좋았고, 무엇이 아쉬웠는지를 KPT로 평가한다.
> Problem과 Try는 반드시 **구체적 개선 행동**으로 귀결된다.
```

일반적인 회고와 다른 점이 두 가지 있습니다:

**첫째, 세 관점으로 분리**합니다:
- **개발 과정**: 기술적 의사결정, 설계, 디버깅 (코드 레벨)
- **AI 협업 과정**: 사용자와 AI 사이의 소통, 맥락 전달 (인터페이스 레벨)
- **워크플로우**: 절차 자체의 완성도 (프로세스 레벨)

이 분리가 중요한 이유는, "이번 세션 잘못됐다"에서 끝나지 않고 **어느 레이어의 문제인지** 진단할 수 있기 때문입니다. 코드가 문제였나, 지시가 문제였나, 절차가 문제였나.

**둘째, Try를 즉시 반영**합니다:

```markdown
## MECE 액션 아이템

| # | 액션 | 카테고리 | 상태 | 긴급도 |
|---|------|---------|------|-------|
| 1 | ...  | 워크플로우 | ❌ | 🔴 즉시 |
```

"다음에 이렇게 하자"로 끝나는 회고가 아닙니다. 🔴 즉시 항목은 **이 세션에서 바로** `.agent/workflows/*.md`나 `.agent/rules.md`를 수정합니다. 수정 결과가 다음 세션부터 즉시 적용됩니다.

```
# 실제 git commits — /retrospect가 워크플로우를 수정한 증거
be8a554a retrospect: DOM 우회 금지 규칙 추가
3d499985 rules: 해결 = 엔트로피 감소. 동작하는 것은 해결이 아니다.
667bdc3a workflow: 설계 문서 없이 코드 수정 금지
b79a9d56 workflow: /go skip 프리셋 제거
```

이 커밋들은 전부 회고에서 발견된 문제를 즉시 반영한 것입니다. "DOM을 직접 건드리지 마"라는 규칙, "동작하는 것은 해결이 아니다"라는 원칙 — 이런 것들이 회고에서 나와 rules.md에 영구 반영되었습니다.

### 메타 순환 — 워크플로우가 워크플로우를 개선합니다

`/retrospect`가 "뭘 고칠지" 발견하면, 그 다음은 뭘까요? **워크플로우 자체를 고칩니다.**

이를 위한 전용 워크플로우들이 있습니다:

- **`/rules`** — rules.md(헌법)를 수정합니다. 회고에서 발견된 원칙이 영구 반영됩니다.
- **`/workflow`** — 워크플로우를 생성/수정하는 메타 워크플로우입니다. `/retrospect`가 "이 절차가 부족하다"를 발견하면 `/workflow`가 고칩니다.
- **`/reframe`** — 워크플로우의 비공식 용어를 표준 프레임워크 용어로 치환합니다. 예를 들어 "다시 생각해봐"를 "Double-loop Learning"으로 리네이밍하면, LLM이 관련 학술 지식을 더 잘 활용합니다.

Biome(린터) 도입과 제거의 사이클도 이 메타 순환의 산물입니다. 2월 8일에 도입(`0611d984`)하고, 설정을 여러 번 조정하다가(`9b5bc04c`), 불필요한 부분을 회고에서 식별하여 정리했습니다. 도구도 회고의 대상입니다.

### `/elicit` — `/discussion`의 역방향 (2026-02-22)

1편에서 `/discussion`은 사용자가 AI에게 "왜?"를 묻는 공간이라고 했습니다. 파이프라인이 성숙한 뒤(02-22), 역방향이 태어났습니다:

```
da7227de elicit: OS 존재 이유 + 산출물 강제 원칙 Codify
d9802ed1 mistakes → precedents 리네임 + /elicit 워크플로우 등록
```

`/elicit`은 **AI가 사용자에게 "왜?"를 묻습니다**. 잘못된 의사결정에서 출발하여, 전문가(사용자)의 암묵지를 체계적으로 추출합니다. "OS의 존재 이유"같은 근본적 질문은 사용자가 먼저 말하지 않습니다 — AI가 물어봐야 나옵니다.

`/discussion`(AI에게 묻기) + `/elicit`(AI가 묻기) = **양방향 Double-loop**. 이 쌍이 완성되기까지 프로젝트 시작으로부터 19일이 걸렸습니다.

### TestBot 진화 — `/retrospect`가 도구 자체를 바꿉니다

회고가 코드나 워크플로우만 바꾸는 게 아닙니다. **테스트 도구 자체**도 회고의 대상입니다:

```
6a476401 02-10 refactor(testBot): restructure with FSD architecture
1fc939bf 02-21 feat(testbot-v2): T1 Vitest Browser Mode — 832/832 tests pass
21d3ea8c 02-21 project: Replay scaffold — TestBot v2를 Replay로 리네이밍
1915ef1d 02-21 feat(testbot): BddReplayPanel — run BDD tests in browser
```

TestBot v1(02-10)은 단순한 테스트 러너였습니다. 회고에서 "테스트 결과를 시각화해야 사람이 검증할 수 있다"는 발견이 나왔고, v2(02-21)에서 Vitest Browser Mode + BDD Replay로 진화했습니다. **도구가 사용자의 검증 능력을 확장**한 것입니다.

## 4. `/auto` — 반쪽짜리 성공

여기까지의 구조를 정리하면 다음과 같습니다:

- **`/go`**: 상태 기반 라우터. 현재 상태를 보고 다음 워크플로우를 호출
- **수동 운영**: 사람이 `/go`를 한 단계씩 눌러줌
- **Knowledge Middleware + /retrospect**: 세션 간 학습

이것이 "작동하지만 귀찮은" 상태였습니다. 최근에 Claude Code에 **Hooks** 기능이 추가되면서, 이 수동 단계를 기계적으로 해결할 수 있게 되었습니다.

커뮤니티에서는 이걸 **Ralph Wiggum 기법**이라고 부릅니다 (Paddo, "Ralph Wiggum: Autonomous Loops for Claude Code"). Stop Hook — 에이전트가 멈추려고 할 때 가로채서 "계속하라"고 지시하는 것입니다.

이 프로젝트에서 구현한 `/auto`는 이렇습니다:

```markdown
## /auto — 자율 파이프라인 실행

> /auto = 마커 ON + /go

1. 마커 파일을 생성한다 → Stop Hook이 활성화됨
2. /go 파이프라인을 그대로 실행한다
3. 워크플로우 완료 시 멈추려 하면 → Stop Hook이 "계속하라"고 지시
4. /archive 단계에서 마커 삭제 → 정상 종료
```

실행하면 이런 일이 벌어집니다:

```bash
touch /tmp/.go-pipeline-active    # 마커 생성
# → /go가 /plan을 실행
# → /plan 완료, 멈추려 함
# → Stop Hook: "마커가 살아있으니 계속 진행하세요"
# → /go가 /project를 실행
# → ...
# → /archive에서 마커 삭제
# → 정상 종료
```

### 왜 "반쪽짜리"인가

솔직하게 말씀드립니다. `/auto`는 **기계적으로 이어 붙이는 것만** 해결했습니다. 아직 해결 못 한 것들이 있습니다:

1. **품질 게이트가 부족합니다** — 각 단계의 산출물을 자동으로 검증하는 메커니즘이 불완전합니다. `/audit`이 "문제 없습니다" 한 줄로 넘어가는 문제가 여전히 발생할 수 있습니다.

2. **자가 진단이 안 됩니다** — 3편에서 말한 "LLM은 자기가 헤매는지 모른다" 문제가 그대로입니다. `/why`를 자동으로 호출하는 트리거가 없습니다.

3. **컨텍스트 소진** — 한 context window 안에서만 유효합니다. 프로젝트가 크면 중간에 컨텍스트가 찰 수 있습니다. 이때 상태 복원은 BOARD.md에 의존합니다.

`/auto`는 "수동 변속기"에서 "반자동 변속기"로 올라간 것입니다. 완전 자동이 아닙니다. 하지만 **워킹은 합니다.** 실제로 이 글을 쓰는 것도 `/auto`로 실행 중입니다.

## 5. 전체 파이프라인 구성도

지금까지 만들어진 파이프라인의 전체 모습입니다:

```
┌─────────────────── /auto (Stop Hook) ───────────────────┐
│                                                          │
│  ┌─────────┐    ┌──────┐    ┌─────────┐                │
│  │/discussion│───→│/plan │───→│/project │                │
│  │ (생각)   │    │(명세) │    │(scaffold)│                │
│  └─────────┘    └──────┘    └────┬────┘                │
│       ↑                          │                      │
│       │ /conflict                ↓                      │
│       │ /blueprint     ┌──── /go 라우터 ────┐           │
│                        │                    │           │
│              ┌────┐  ┌─────┐  ┌──────┐  ┌───────┐     │
│              │/spec│→│/red │→│/green│→│/verify│     │
│              │(정의)│  │(스펙)│  │(구현) │  │(검증)  │     │
│              └────┘  └─────┘  └──────┘  └───┬───┘     │
│                                              │          │
│                              ┌───────────────┘          │
│                              ↓                          │
│                    ┌──────────────────┐                 │
│                    │  다음 Task 있음?  │                 │
│                    └───────┬──────────┘                 │
│                     Yes ↓       No ↓                    │
│                    루프 ←   ┌───────┐                   │
│                             │/audit │                    │
│                             │/doubt │                    │
│                             └───┬───┘                   │
│                                 ↓                       │
│                    ┌────────────────────┐               │
│                    │/retrospect→/archive│               │
│                    └────────────────────┘               │
│                              ↓                          │
│                         마커 삭제                        │
└─────────────────────────────────────────────────────────┘

복구 경로 (어느 단계에서든 삽입 가능):
  /why     — 막혔을 때 → 근본 원인 보고서
  /reflect — 매 단계 후 → 방향 점검
  /redteam — 설계 완료 후 → 반박 분석

지식 순환 (모든 단계에 암묵 적용):
  _middleware — 시작: 지식 로딩 / 실행 중: 발견 누적 / 종료: 영구 반영
```

## 6. 이상향 — 완전 자율 품질 보장 루프

지금의 파이프라인이 최종이 아닙니다. 이상향은 이렇습니다:

```
[현재]
/auto → 기계적 이어붙이기 + 수동 품질 판단

[이상향]
/auto → 각 단계에서 자동 품질 게이트 + 자가 진단 + 자동 복구
```

구체적으로 필요한 것은 다음과 같습니다:

| 현재 | 이상향 |
|------|--------|
| /audit이 "문제 없습니다"로 넘어갈 수 있다 | 산출물 존재 + 형식 검증을 자동화 |
| /why를 사람이 호출해야 한다 | 반복 패턴 감지 → 자동 /why 트리거 |
| 컨텍스트 소진 시 수동 재개 | BOARD.md 기반 자동 복원 + 이어가기 |
| 지식 반영을 AI가 판단 | 지식 반영 형식 검증 자동화 |

하지만 근본적 한계가 있습니다: **LLM은 자기 출력의 품질을 자기가 평가할 수 없습니다.** 이건 Godel의 불완전성 정리와 같은 구조적 한계입니다. 그래서 완전 자율은 불가능하고, **"사람의 개입을 최소화하되 0으로 만들지는 않는"** 것이 현실적 목표입니다.

## 7. 현재 위치 — 여기까지 왔고, 여기를 갑니다

이 시리즈 전체를 한 장으로 요약합니다:

```
"계획을 세우세요" (남의 말)
        ↓
  안 함. 바로 시킴.
        ↓
  Goal Fixation — LLM이 수정 본능으로 폭주
        ↓
  "왜?" → 산출물 재정의 → /discussion (생각/실행 분리)
        ↓
  문서 + TDD + BOARD (세 기둥)
        ↓
  한 턴의 함정 → /red + /green (워크플로우 쪼개기)
        ↓
  산출물 중심 사고 → 워크플로우 폭발 (47개)
        ↓
  /go (자동 라우터) → 한 세션 함정 → 귀찮지만 하나씩
        ↓
  폭주 → Task 분해 = 브레이크 포인트 → /divide
        ↓
  복구 경로 — /why, /reflect, /redteam
        ↓
  자가 학습 — _middleware + /retrospect
        ↓
  /auto (Stop Hook) — 반쪽짜리 성공
        ↓
  "계획을 세우세요" (내 경험)
```

원점으로 돌아왔습니다. 하지만 같은 자리가 아닙니다.

처음의 "계획을 세우세요"는 남의 조언이었습니다. 지금의 "계획을 세우세요"는 **폭주를 겪고, 파이프라인을 만들고, 실패하고, 다시 만든 끝에 도달한 결론**입니다.

그리고 이 결론도 완성이 아닙니다. `/auto`는 반쪽짜리 성공이고, 품질 게이트는 미완성이고, 자가 진단은 아직 먼 이야기입니다. **가는 중입니다.**
