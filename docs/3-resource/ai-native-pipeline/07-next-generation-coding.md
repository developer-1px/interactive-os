# 다음 세대의 코딩: 비결정적 리턴의 시대

*2026년, 소프트웨어 공학의 60년 전제가 흔들리고 있다.*

---

모든 개발자가 첫날 배우는 것이 있다. `add(2, 3)`은 `5`를 돌려준다. 내일도, 모레도, 서버가 바뀌어도. 이것이 함수다. 이것이 프로그래밍이다.

그런데 지금, 수백만 개발자가 매일 쓰는 도구가 이 전제를 깨고 있다. Cursor, Claude Code, Devin, GitHub Copilot — 이들은 모두 같은 입력에 다른 출력을 돌려준다. 어제 깔끔하게 나온 코드가 오늘은 엉망이다. 버그가 아니다. **이것이 설계다.**

LLM은 비결정적 함수다. 그리고 이 비결정적 함수를 일상적으로 호출하게 된 순간, 코딩이라는 행위 자체가 변하기 시작했다.

---

## 60년간 당연했던 것

1960년대 Fortran부터 2020년대 TypeScript까지, 프로그래밍 언어의 세대가 바뀌어도 하나의 전제는 불변이었다. **같은 입력이면 같은 출력**. 이 위에 타입 시스템이 올라갔고, 단위 테스트가 올라갔고, CI/CD가 올라갔고, 정적 분석이 올라갔다. 소프트웨어 공학의 60년 도구 체인 전체가 이 하나의 가정 위에 세워져 있다.

`temperature=0`으로 설정하면 LLM도 결정적으로 동작할까? Thinking Machines Lab(2025)의 연구에 따르면, 그렇지 않다. GPU의 부동소수점 연산 순서, 배치 크기, 병렬 처리 스케줄링 — 이런 하드웨어 수준의 변수들이 동일 입력에서도 서로 다른 출력을 만든다. 이 팀은 "batch-invariant inference"로 재현성을 달성했다고 보고했지만, 이는 추론 인프라의 해법이지 모델 자체를 결정적으로 만든 것은 아니다.

문제는 이것이다: **개발자의 일상 도구가 비결정적 함수가 되었는데, 도구 체인은 여전히 결정적 패러다임 위에 있다.**

---

## "Prompt Chaining"이라고 불러야 할까, "함수 합성"이라고 불러야 할까

Anthropic은 2024년 [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)에서 에이전트 시스템의 핵심 원칙을 발표했다. 거기서 그들이 제안한 첫 번째 패턴이 **Prompt Chaining**이다 — 복잡한 작업을 작은 단위로 쪼개고, 각 단위의 출력을 다음 단위의 입력으로 연결한다.

이 설명을 읽고 기시감을 느낀 개발자가 있을 것이다. 이것은 **함수 합성(function composition)**이다. `pipe(f, g, h)`. Unix 파이프라인. 마이크로서비스 오케스트레이션. 50년 전부터 있던 것이다.

차이는 하나다. `f`, `g`, `h`가 비결정적이라는 것.

agentic-design.ai(2025)의 연구에 따르면, Prompt Chaining은 monolithic 프롬프트 대비 정확도를 15.6% 향상시킨다. 왜? 비결정적 함수의 "오차 반경"을 줄이기 때문이다. 큰 함수 하나가 낮은 확률로 정답을 맞추는 것보다, 작은 함수 여러 개가 각각 높은 확률로 부분 정답을 맞추는 편이 전체 정확도가 높다.

소프트웨어 공학은 이미 이 패턴의 이름을 알고 있다. Bertrand Meyer(1992)는 이것을 **Design by Contract**라 불렀다 — 함수의 전제조건, 사후조건, 불변조건을 명시하여 호출 범위를 제한하는 기법. LLM 시대에 이것은 "이 프롬프트가 받아야 할 입력", "이 프롬프트가 만들어야 할 산출물", "이 프롬프트가 건드리면 안 되는 영역"으로 번역된다.

업계는 이것을 **스킬(Skill)**이라 부르기 시작했다. 계약이 씌워진 비결정적 함수. 결정적 함수를 function이라 부르듯, 비결정적 함수를 skill이라 부르는 것이다.

---

## `pipe(f, g, h)`에서 `pipe(f, gate, g, gate, h)`로

결정적 함수의 합성은 자유롭다. `pipe(parse, validate, transform)` — 중간에 아무것도 필요 없다. 각 함수가 항상 같은 출력을 돌려주므로, 합성 결과도 결정적이다.

비결정적 함수의 합성은 위험하다. 1단계에서 발생한 미세한 오류가 2단계에서 증폭되고, 3단계에서 파국이 된다. Alvarez & Marsal(2025)은 이를 다음과 같이 요약한다:

> *"Chaining multiple AI-driven steps can compound these issues if not properly guarded."*

해법은 **게이트 합성(gated composition)**이다. 각 함수 사이에 검증 관문(gate)을 배치하여, 사후조건이 충족되었을 때만 다음 함수를 호출한다. Robert Cooper(1990)의 Stage-Gate System이 제품 개발에서 한 것을, LLM 파이프라인에서 반복하는 것이다.

이 게이트 합성 구조를 업계에서는 **하네스(Harness)**라 부른다. nxcode.io(2025)는 이를 **"harness engineering"**이라 명명했다 — AI 에이전트의 환경, 제약, 피드백 루프를 설계하는 분야.

핵심 통찰은 이것이다: 결정적 함수 시대의 코딩은 **함수를 작성하는 것**이었다. 비결정적 함수 시대의 코딩은 **함수 사이의 게이트를 설계하는 것**이다. 코딩의 무게 중심이 **구현에서 검증으로** 이동하고 있다.

---

## AI는 자기 코드를 검증할 수 없다

여기에 한 가지 불편한 사실이 있다.

Snorkel AI(2025)의 실험에서, 최신 LLM 모델에게 자기 출력을 비판(self-critique)하게 했더니 정확도가 98.1%에서 56.9%로 떨어졌다. 모델이 올바른 출력을 "틀렸다"고 판정하고 수정한 것이다.

이것은 소프트웨어 테스팅의 **오라클 문제(Oracle Problem)**와 정확히 일치한다. Barr et al.(*IEEE TSE*, 2015)이 정의한 이 문제는, "시스템의 관찰된 동작이 올바른지를 어떻게 판단하는가?"라는 근본 질문이다. 결정적 함수에서 오라클은 단순하다 — `expect(add(2, 3)).toBe(5)`. 비결정적 함수에서 오라클은 근본적으로 다르다 — 기대값이 없기 때문이다.

BEAVER(arxiv, 2025)는 이 문제의 한 가지 해법을 제시한다. LLM 출력에 대해 결정적 확률 경계(deterministic probability bounds)를 계산하는 프레임워크다. 핵심 메시지는 명확하다: **검증은 반드시 외부 결정적 시스템에서 이루어져야 한다.** 컴파일러, 테스트 러너, 정적 분석기 — 이것들이 비결정적 시대의 진짜 안전장치다.

"vibe coding"이라는 용어가 있다. AI에게 대략적인 의도를 전달하고, 나온 코드를 그대로 쓰는 방식이다. The New Stack(2025)은 이를 "vibe coding hangover"라 부르며, 2026년에 순수 AI 생성 애플리케이션의 실패가 본격화될 것으로 예측한다. MIT(2025)의 연구에서 기업 AI 프로젝트의 95%가 의미 있는 비즈니스 가치를 전달하지 못한다는 결과가 나왔다. 원인 중 하나: 검증 루프의 부재.

## 기억하지 못하는 함수

비결정적 함수의 또 다른 특수성이 있다. **이 함수는 학습하지 않는다.**

LLM은 순수 함수(pure function)다. 입력(컨텍스트)에서 출력을 생성하되, 자기 가중치를 변경하지 않는다. 세션 N에서 "이 라이브러리 함수는 함정이다"를 발견해도, 세션 N+1에서는 같은 함정에 빠진다. 매 세션이 Day 1이다.

이 문제에 대한 업계의 대응은 수렴하고 있다:

- **Claude Code**: `CLAUDE.md` 파일과 structured note-taking (Anthropic, 2025)
- **Cursor**: `.cursor/rules/` 폴더 (project-level rules)
- **GitHub Copilot**: `agents.md` (agent role definition)
- **Devin**: 레포지토리별 knowledge base

모두 같은 구조다. LLM의 가중치(함수 정의)를 바꿀 수 없으니, **입력에 포함되는 컨텍스트(환경)를 바꾼다.** 함수형 프로그래밍에서 클로저가 자유 변수를 캡처하는 것과 동일한 메커니즘이다.

Anthropic은 이 분야를 **"context engineering"**이라 명명했다. 2025년 말, Linux Foundation이 Agentic AI Foundation을 설립하면서 에이전트 생태계의 표준화가 시작되었다. 그 핵심 의제 중 하나가 바로 에이전트의 기억 관리 표준이다.

## 다음 세대의 코딩은 무엇이 다른가

정리하면 이렇다.

**이전 세대의 코딩:**
- 함수를 작성한다
- 함수는 결정적이다 (같은 입력 → 같은 출력)
- 테스트는 기대값과 비교한다 (`expect(f(x)).toBe(y)`)
- 함수를 자유롭게 합성한다 (`pipe(f, g, h)`)
- 실패하면 스택 트레이스를 읽는다

**다음 세대의 코딩:**
- **스킬을 작성한다** — 계약(전제조건, 사후조건, 불변조건)이 씌워진 비결정적 함수
- **리턴이 비결정적이다** — 같은 입력에 다른 출력. "정답"이 아니라 "속성"을 만족하면 통과
- **테스트는 속성을 검증한다** — Property-Based Testing. `expect(f(x)).toSatisfy(property)`
- **게이트 합성한다** — `pipe(f, gate, g, gate, h)`. 검증 없이 다음 단계로 넘어가지 않는다
- **실패해도 에러가 없다** — LLM은 자기가 틀린 줄 모른다. 외부 검증기가 유일한 안전장치다
- **함수가 기억을 갖는다** — 외부 메모리가 세션 간 지식을 보존한다

| 개념 | 결정적 시대 | 비결정적 시대 |
|------|-----------|-------------|
| 기본 단위 | function | **skill** (contract-bound nondeterministic function) |
| 리턴 타입 | `T` | `T \| Error \| Hallucination` |
| 합성 | `pipe(f, g, h)` | `pipe(f, gate, g, gate, h)` |
| 테스트 | `assert(f(x) === y)` | `assert(f(x) satisfies P)` |
| 오라클 | 기대값 | 외부 결정적 검증기 |
| 기억 | 변수, DB | exocortex (외부 메모리 + context engineering) |
| 실패 모드 | exception | silent hallucination |
| 안전장치 | `try/catch` | gate + circuit breaker |

## 코딩이 변하고 있다

역사적으로 프로그래밍 패러다임의 전환은 **새로운 종류의 복잡성**에 대한 대응이었다. 구조적 프로그래밍은 goto의 복잡성에 대응했고, 객체지향은 상태 관리의 복잡성에 대응했고, 함수형은 부수효과의 복잡성에 대응했다.

비결정적 함수 패러다임은 **출력의 불확실성에 대한 대응**이다. 처음으로, 프로그래머가 호출한 함수의 리턴값을 신뢰할 수 없는 세계에서 코딩하는 방법을 체계화하는 것이다.

이 패러다임은 아직 이름이 없다. 도구도 표준도 미성숙하다. 하지만 패턴은 이미 수렴하고 있다 — Design by Contract, Stage-Gate, Circuit Breaker, Property-Based Testing, Transactive Memory. 모두 이미 존재하는 패턴이다. 새로운 것은 없다. 다만, 이 패턴들이 **동시에, 같은 문제를 위해, 같은 구조로** 결합되고 있다는 것이 새롭다.

다음 세대의 코딩에서 가장 중요한 스킬은 코드를 작성하는 것이 아닐 수 있다. **비결정적 리턴을 가진 함수를 안전하게 합성하는 것** — 그것이 이 시대의 코딩이다.
