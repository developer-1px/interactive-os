# 2편: 산출물이 행동을 결정한다

> AI Native 파이프라인: /auto까지의 여정 (2/4)
>
> [이전 편: 1편 — "계획을 세우세요" (근데 왜 어려울까?)](./01-why-planning-is-hard.md)

## 이전 편 요약

1편에서 발견한 것: LLM에게 "하지 마"는 안 통합니다. 대신 **산출물을 재정의하면 행동이 바뀝니다**. 이 원리로 `/discussion`이 탄생했습니다.

이번 편에서는 이 "산출물 중심 사고"가 어떻게 확장되고, 어디서 부딪히고, 무엇을 만들어냈는지를 다룹니다.

---

## 1. 코드를 안 보게 됩니다

`/discussion`으로 "생각하는 법"을 만들었습니다. 다음 문제는 "실행"이었습니다. LLM이 코딩하니까 생산성은 올라갔는데, 부작용이 있었습니다:

> **인간이 코드를 한 줄씩 안 읽게 됩니다.**

이건 자연스러운 결과입니다. LLM이 100줄을 짜면, 사람은 "되나 안 되나"만 확인합니다. 하지만 "된다"고 해서 "맞다"는 아닙니다. 동작하지만 유지보수가 불가능한 코드, 동작하지만 기존 패턴과 어긋나는 코드가 쌓입니다.

초기에는 Biome(린터)을 도입해서 도구로 품질을 잡으려 했습니다 (2월 8일, `0611d984`). 하지만 린터는 "코드 스타일"을 잡을 뿐, "방향이 맞는가"는 답해주지 못합니다. 도구만으로는 부족했습니다.

그래서 세 가지가 동시에 필요해졌습니다:

| 문제 | 왜 필요한가 | 해법 |
|------|-----------|------|
| 코드를 안 본다 | "맞다"를 사람이 검증할 수 없다 | **TDD** — 테스트가 유일한 품질 증명 |
| 기억이 날아간다 | 지난 세션에서 뭘 했는지 모른다 | **문서** — 세션 간 메모리 |
| 어디까지 했는지 모른다 | 같은 일을 반복하거나 빠뜨린다 | **진척도 추적** — 상태 머신 |

이 셋은 독립이 아니라 **서로를 강화**합니다. 테스트가 있으니 "뭐가 되고 안 되는지" 문서 없이도 알 수 있고, 문서가 있으니 "왜 이렇게 짰는지" 다음 세션이 알고, 진척도가 있으니 "다음에 뭘 해야 하는지" 세션이 바뀌어도 이어갈 수 있습니다.

여기에 하나 더 — 세 기둥 이전에 **전제조건**이 있습니다. 개발 환경이 깨져있으면 세 기둥이고 뭐고 삽질만 합니다. 그래서 `/ready`가 생겼습니다 — 세션 시작 시 서버, 타입 체크, 렌더 스모크까지 확인하는 프리플라이트 체크입니다. 2월 14일 "헌법화" 당일에 함께 만들어졌습니다.

## 2. 한 턴의 함정 — `/tdd`의 실패

TDD가 필요하다는 건 알았습니다. 그래서 `/tdd`라는 워크플로우를 만들었습니다. "테스트를 먼저 쓰고, 구현하라."

**이건 완전히 실패했습니다.**

왜일까요? LLM에게는 **"한 턴"**이라는 개념이 있습니다. 사람의 TDD는 이렇습니다:

```
Red 작성 → (시간이 흐른다) → 실패를 보고 설계를 고민한다 → Green 작성
```

중간에 시간이 흐르는 것이 핵심입니다. 실패한 테스트를 보면서 "이걸 어떻게 통과시키지?"를 고민하는 과정에서 설계가 나옵니다.

LLM의 TDD는 이렇습니다:

```
Red 작성 → (같은 턴에서) → 이미 답을 알고 있으니 바로 Green 작성
```

한 턴 안에서 Red→Green을 하면, 테스트는 "스펙"이 아니라 **"구현의 거울"**이 됩니다. 구현을 먼저 머릿속에 짜고, 그거에 맞는 테스트를 역으로 씁니다. 순서만 TDD고 **효과는 0**입니다.

더 심한 경우도 있었습니다:

```
AI: "테스트 코드가 현재 구조와 맞지 않아 불필요합니다. 제거하고 구현을 수정하겠습니다."
```

테스트를 삭제해버립니다. Goal Fixation(1편 참조)이 작동한 것입니다 — "코드를 동작시키는 것"이 목표이니, 테스트가 방해되면 테스트를 지우는 게 "합리적"입니다.

## 3. 워크플로우를 쪼개야 합니다

`/tdd`의 실패에서 깨달은 것이 있습니다:

> **"하지 마"는 안 통하지만, "이것만 해"는 통합니다.**

프롬프트로 "구현하지 마, 테스트만 써"라고 해도 밀립니다. 하지만 **워크플로우 자체를 두 개로 쪼개면** 물리적으로 불가능합니다.

그래서 `/tdd` 하나를 `/red`와 `/green` 두 개로 분리했습니다. 실물을 보겠습니다:

### `/red` — 실패하는 테스트만 작성

```markdown
## /red — 실패하는 테스트 작성

> **산출물 1개**: 테스트 코드 `.test.ts` (🔴 FAIL)
> **금지**: 구현 코드 작성. 이 세션에서 `src/` 프로덕션 코드를 수정하지 않는다.
> **원칙**: 테스트가 스펙이다. 구현 방법을 모르는 상태에서 **기대 동작만** 기술한다.
```

첫 세 줄로 범위가 완벽하게 제한됩니다:
- 산출물: `.test.ts` 파일 하나
- 금지: 프로덕션 코드 수정
- 원칙: 기대 동작만. 구현 방법은 몰라도 됩니다.

### `/green` — 테스트를 통과시키는 최소 구현

```markdown
## /green — 테스트를 통과시킨다

> **산출물**: 구현 코드 (🟢 PASS)
> **전제**: `/red`에서 작성한 🔴 FAIL 테스트가 존재해야 한다.
> **원칙**: 테스트가 요구하는 것만 구현한다. 과하게 구현하지 않는다.
```

`/green`에는 **전제 조건**이 있습니다: Red 테스트가 이미 존재해야 합니다. 테스트 없이는 시작조차 못 합니다.

### 왜 분리가 작동하는가

| `/tdd` (하나) | `/red` + `/green` (분리) |
|--------------|------------------------|
| 한 턴에 Red+Green | 물리적으로 다른 세션 |
| 답을 이미 알고 테스트 작성 | 구현 모르는 상태에서 테스트 작성 |
| 테스트 = 구현의 거울 | 테스트 = 진짜 스펙 |
| 테스트 삭제 가능 | 테스트 삭제 불가 (다른 세션에서 작성됨) |

핵심은 **세션을 물리적으로 끊는 것**입니다. `/red` 세션이 끝나면, 그 테스트 파일은 git에 커밋됩니다. `/green` 세션은 그 테스트를 **입력으로 받습니다**. 테스트를 삭제하려면 의도적으로 git revert를 해야 하는데, 그건 Goal Fixation의 범위를 넘어섭니다.

이건 제조업의 **Quality Gate**(Stage-Gate) 개념과 같습니다. Robert Cooper가 제품 개발에서 제안한 것으로, 각 공정 단계를 통과할 때마다 기준을 충족해야 다음으로 넘어갈 수 있습니다 (Cooper, "Stage-Gate Systems", *Business Horizons*, 1990). 완성품을 검사하는 게 아니라, **각 단계에서** 검사합니다.

```
# 실제 git log — /red와 /green이 분리된 세션으로 동작한 증거
1e11ca75 docs: spec.md — headless-simulator 성공 기준 5가지
9306578c docs: T1 실행 전략 — 산출물 3종 (코드, e2e GREEN, vitest 동등)
    ↕ (세션 경계)
2ea061f1 fix(T1): 25/29 e2e PASS — tree expandable + DOM fallback 완성
e0bcc6d6 fix(T1): 29/29 e2e ALL GREEN
```

spec → red → green이 별도 커밋으로 나뉘어 있습니다. 한 세션에 우겨넣은 게 아닙니다.

## 4. 파이프라인이라는 발견

`/red`와 `/green`을 분리하면서 더 큰 발견이 있었습니다:

> **워크플로우 자체가 산출물입니다.**

처음에는 "산출물 = 코드 or 문서"였습니다. 하지만 `/red` → `/green`이라는 **연결 자체**가 산출물입니다. 각 워크플로우가 산출물을 내고, 그 산출물이 다음 워크플로우의 입력이 됩니다.

| 워크플로우 | 산출물 (출력) | 다음 단계의 입력 |
|-----------|-------------|----------------|
| `/discussion` | Knowledge + Claim | `/plan`의 목표 |
| `/plan` | 변환 명세표 | `/project`의 태스크 |
| `/usage` | 컴파일 안 되는 이상적 코드 | `/spec`의 기대 동작 |
| `/red` | 🔴 FAIL 테스트 | `/green`의 전제 조건 |
| `/green` | 🟢 PASS 코드 | `/bind`의 입력 |
| `/bind` | UI에 연결된 코드 | `/verify`의 검증 대상 |

이건 Unix 철학의 `pipe`와 같습니다: `cat | grep | sort`. 각 단계는 자기 일만 하고, 연결이 전체를 만듭니다. 산출물이 파이프를 타고 흐릅니다.

여기서 관리 대상이 바뀝니다:

| 단계 | 관리 대상 | 사람의 역할 |
|------|----------|------------|
| 초기 | 코드 | 코드를 직접 쓴다 |
| 중기 | 산출물 (코드+문서+테스트) | 산출물을 지시한다 |
| **현재** | **파이프라인** | **"각 단계의 산출물이 나왔는가?"만 확인** |

코드를 관리하는 게 아니라, **"이 단계에서 이 산출물이 존재하는가?"**를 관리합니다. 코드를 안 읽어도 되는 이유가 여기서 완성됩니다 — 파이프라인을 통과했으면 품질이 보장되니까요.

## 5. 산출물 중심 사고의 폭발

"이 상황의 산출물은 뭐지?"를 먼저 묻기 시작하자, 모든 문제에서 워크플로우가 하나씩 생겨났습니다:

| 상황 | 산출물 | 워크플로우 |
|------|--------|----------|
| 방향이 안 정해졌다 | 구조화된 합의 | `/discussion` |
| 뭘 해야 할지 모른다 | 변환 명세표 | `/plan` |
| 이상적 API를 먼저 그리고 싶다 | 컴파일 안 되는 Usage 코드 | `/usage` |
| 기능 정의가 필요하다 | BDD 시나리오 | `/spec` |
| 테스트가 필요하다 | 실패하는 테스트 | `/red` |
| 커버리지 갭이 보인다 | 갭 분석 리포트 | `/coverage` |
| 구현이 필요하다 | 통과하는 코드 | `/green` |
| headless → UI 연결 | 화면에서 동작하는 UI | `/bind` |
| 코드가 올바른지 확인 | 감사 보고서 | `/audit` |
| 철학·네이밍 준수 확인 | 리뷰 리포트 | `/review` |
| 과잉인지 확인 | 의심 보고서 | `/doubt` |
| 큰 문제를 쪼개야 한다 | Work Package 트리 | `/divide` |
| 설계를 검증해야 한다 | Red Team 분석 | `/redteam` |
| 성능 문제 | 측정→진단→수정 보고서 | `/perf` |
| 코드 전에 보고 싶다 | HTML+Tailwind 프로토타입 | `/design` |
| 사용자 관점이 필요하다 | 유저 스토리 목록 | `/stories` |
| LLM 산출물이 깨졌다 | 교정된 산출물 | `/fix` |
| 세션을 마무리한다 | KPT 회고 | `/retrospect` |

이걸 **한번에 설계한 게 아닙니다**. 필요에 의해 하나씩 생겨났습니다. "여기서 산출물이 끊긴다"는 경계가 보일 때마다 워크플로우가 생겼습니다.

이 프로젝트의 git history를 보면 그 흔적이 보입니다:

```
# 2026-02-11: 최초 워크플로우 체계화 시작
62800a38 discussion: headless zone architecture + add reflect/retrospect workflows

# 2026-02-14: 하루에 22개 워크플로우 일괄 정비 — "헌법화"
7be8bb93 docs: rules.md v2 — 선언문 헌법화

# 2026-02-15: /doubt 생성 → 즉시 5개 불필요 워크플로우 제거
df11f66b refactor(workflows): /doubt 적용 — 35→30 워크플로우 축소

# 2026-02-18: /solve 생성 — 7개 프레임워크를 한 워크플로우로 통합
```

2월 14일에서 18일 사이가 **폭발적 성장기**입니다. 일주일 동안 워크플로우 인프라의 60%가 만들어졌습니다. 이건 Domain-Driven Design에서 Bounded Context가 발견되는 과정과 같습니다 — 실제 문제를 풀다 보면 경계가 드러납니다.

## 6. BOARD.md — 파이프라인의 상태 머신

워크플로우가 47개까지 늘어났습니다. 파이프라인이 있고, 각 단계가 산출물을 내놓습니다. 그런데 한 가지 문제가 남아 있습니다:

> **"어디까지 했는지"를 어떻게 추적합니까?**

Agile에서는 이걸 **Information Radiator**라고 부릅니다. 벽에 칸반 보드를 붙여놓으면 지나가다 보이는 것처럼, 프로젝트의 현재 상태가 항상 "방사"되어야 합니다 (Alistair Cockburn, *Agile Software Development*, 2001).

LLM에게 이건 더 중요합니다. 사람은 기억이라도 하지만, **LLM은 진짜로 매 세션 0에서 시작**하니까요.

이 프로젝트에서 진척도를 추적하는 방법은 `BOARD.md` 파일입니다. 실물을 보겠습니다:

```markdown
# headless-simulator

## Context
Claim: TestPage가 DOM 없이 Playwright 수준의 검증을 달성한다.
Before → After: Playwright 의존 → headless 단독 검증
Risks: DOM 없이 접근성 속성을 검증하면 실제 브라우저와 괴리 가능

## Now
- [ ] T3: triggerRegistry — headless trigger 시뮬레이션
- [ ] T4: zoneContext — 포커스/활성화 컨텍스트 에뮬레이션

## Done
- [x] T1: 29/29 e2e ALL GREEN — tsc 0 | +29 tests | build OK
- [x] T2: APG tests migrated to createOsPage — tsc 0 | +15 tests

## Unresolved
- Tooltip 패턴: hover 이벤트를 headless로 시뮬레이션할 수 있는가?
```

이 파일이 하는 일은 다음과 같습니다:

1. **Context**: 이 프로젝트가 뭐고 왜 하는지 (새 세션 온보딩)
2. **Now**: 지금 해야 할 일 (라우팅)
3. **Done**: 증빙이 있는 완료 목록 (증명)
4. **Unresolved**: 아직 답이 없는 질문 (경고)

새 세션이 시작되면 `/go`가 이 파일을 읽고, Now에서 첫 번째 미완료 태스크를 찾아서 실행합니다. 사람이 "이거 해"라고 매번 말할 필요가 없습니다.

Done 항목에 주목해주세요: `tsc 0 | +29 tests | build OK`라는 증빙이 붙어 있습니다. **증빙 없이 완료 표시하는 것은 금지**입니다. 이것이 이 프로젝트의 BOARD.md 규칙입니다.

```
# 실제 git commit — BOARD.md가 진척도를 추적한 증거
e0bcc6d6 fix(T1): 29/29 e2e ALL GREEN
e2e10eac fix(T1): autoFocus resolveItems + T1 완료
f1a98cfa fix(T1): tree toggle test — click activates+expands
2ea061f1 fix(T1): 25/29 e2e PASS — tree expandable + DOM fallback 완성
```

이 커밋 메시지들은 BOARD.md의 T1이 점진적으로 완성되는 과정을 보여줍니다. 25/29 → 29/29. 이것이 진척도 추적입니다.

### 실전 증거: tsc 226→0 (하루)

BOARD.md와 `/verify` 게이트가 실제로 작동한 가장 극적인 사례가 있습니다. 2026년 2월 26일, 하루 만에 TypeScript 에러 226개를 0으로 줄였습니다:

```
# 2026-02-26 — 하루의 기록 (12+ 커밋)
b7f1c72a fix: TS6133 unused vars/imports 31→0 (tsc total 226→199)
a21941b8 fix: TS7006 implicit any 14→0 (tsc total 199→185)
babfafe0 fix: primitives.ts TS4111/2412/2322 (tsc 185→174)
  ... (중간 8개 커밋) ...
b639a4a1 fix: toast types, defineApp EOPT (tsc 13→5)
16de11cc fix: final tsc errors — all 226→0
```

이 뒤를 이어 같은 날 `any` 캐스트 100건도 제거했습니다:

```
60bfae33 refactor: remove ~50 'no-explicit-any' from collection module
fce61359 refactor: remove 16 'no-explicit-any' from playwright shim
  ... (5개 커밋) ...
5668e075 refactor: remove no-explicit-any from inspector and command-palette
```

이것이 가능했던 이유는 두 가지입니다. 첫째, BOARD.md에 "tsc 0"이 증빙 기준으로 명시되어 있으니 **"대충 넘어가기"가 불가능**합니다. 둘째, rules.md에 "`any` 무관용" 원칙이 있으니 타입 에러를 `as any`로 회피하는 **관성적 해법이 차단**됩니다. 원칙(rules.md)과 추적(BOARD.md)이 합쳐져야 이런 결과가 나옵니다.

## 여기까지의 교훈

| 문제 | 발견 | 해법 |
|------|------|------|
| 코드를 안 보게 된다 | 테스트가 유일한 품질 증명 | TDD + BOARD.md 증빙 |
| LLM의 TDD는 TDD가 아니다 | 한 턴에 Red+Green = 효과 0 | `/red`와 `/green` 물리적 분리 |
| "하지 마"는 안 통한다 | "이것만 해"는 통한다 | 워크플로우 = 범위 제한 |
| 산출물을 연결하면 파이프라인 | 관리 대상 = 코드 → 과정 | 각 단계의 산출물 존재 여부가 품질 |
| 어디까지 했는지 모른다 | 세션이 바뀌면 상태가 사라진다 | BOARD.md = LLM을 위한 칸반 |

**핵심 한 줄**: 워크플로우의 첫 번째 설계 질문은 **"이 상황의 산출물은 무엇인가?"**입니다. 산출물이 정해지면 행동이 따라옵니다.
