## 이전 편 요약

2편에서 발견한 것: 워크플로우의 첫 번째 설계 질문은 "이 상황의 산출물은 무엇인가?"입니다. 산출물을 정의하면 행동이 따라오고, 산출물을 연결하면 파이프라인이 됩니다.

## 1. "알아서 해줘" — `/go`의 탄생

워크플로우가 늘어나자 새로운 피로가 생겼습니다:

```
나: /discussion
(합의 도달)
나: /plan
(명세표 작성)
나: /red
(테스트 작성)
나: /green
(구현)
나: /verify
(검증)
...
```

매번 사람이 다음 워크플로우를 호출해야 합니다. 워크플로우 하나하나는 잘 동작하는데, **연결은 여전히 수동**입니다.

그래서 `/go`를 만들었습니다. "시작해"라고 치면, 현재 상태를 파악하고 알맞은 워크플로우를 자동으로 호출하는 **라우터**입니다.

실물을 보겠습니다. `/go`의 핵심은 이 판별표입니다:

```markdown
## 파이프라인 (위에서부터 순서대로 판별)

| # | 판별                  | 라우팅    |
|---|-----------------------|----------|
| 0 | /plan 명세표 없음      | → /plan  |
| 1 | BOARD.md 없음         | → /project |
| 2 | spec.md 없음          | → /spec  |
| 3 | Red 테스트 없음        | → /red   |
| 4 | Red 테스트 FAIL        | → /green |
| 5 | 태스크 완료            | → /refactor |
| 6 | 다음 태스크 있음        | → #2로 루프 |
| 7 | 모든 Now Done         | → /audit → /doubt |
| 8 | audit+doubt 통과      | → /retrospect → /archive |
```

위에서부터 순서대로 조건을 체크하고, **첫 번째 매칭에서 실행**합니다. 한 단계가 끝나면 다시 표의 맨 위부터 판별합니다. 사람이 `/go`만 치면, 계획부터 회고까지 전체 사이클을 자율 실행합니다.

```
# 실제 git commit — /go가 재설계된 시점
31b68522 feat: /go 4-Phase 재설계 + OS RUNBOOK 신설

# 이전에는 /go가 Phase 1~4를 순서대로 실행하는 구조였으나,
# 판별표 방식으로 바뀌면서 "상태 기반 라우팅"이 되었다.
```

### 그런데 망했습니다

`/go`를 실행하면, 모든 스킬을 한 세션 안에 우겨넣어서 해버렸습니다. `/tdd`를 하나로 만들었을 때와 **같은 문제가 한 단계 위에서 재발**한 것입니다:

```
[/tdd의 한 턴 함정]
Red 작성 → (같은 턴에서) → Green 작성 → 효과 0

[/go의 한 세션 함정]
plan → spec → red → green → audit → (같은 세션에서) → 다 대충 → 효과 0
```

앞부분은 디테일하게 하다가 뒤로 갈수록 대충 넘어갑니다. audit은 "문제 없습니다" 한 줄로 끝나고, doubt는 "과잉 없습니다"로 넘어갑니다. **한 턴의 함정이 프랙탈처럼 스케일만 바꿔서 재발**한 것입니다.

## 2. 정직한 저점 — "귀찮지만 하나씩 누르자"

여기서 정직해져야 합니다. **자동화를 포기했습니다.**

`/go`를 한번에 돌리면 품질이 무너집니다. 하지만 `/go`가 각 단계를 자동 호출하는 대신, **사람이 한 단계씩 "다음" 버튼을 누르면** 각 워크플로우가 제대로 동작합니다.

```
나: /go
(plan 실행됨)
— 세션 끝 —

나: /go
(spec 실행됨)
— 세션 끝 —

나: /go
(red 실행됨)
— 세션 끝 —
```

귀찮습니다. 매번 `/go`를 치고, 결과를 확인하고, 다시 `/go`를 칩니다. 하지만 **수동으로 눌러도 파이프라인의 가치는 유지**됩니다:

- 사람이 하는 건 **"다음"을 누르는 것**뿐
- 각 단계가 **뭘 해야 하는지**는 워크플로우가 알고 있습니다
- 산출물이 **나왔는지 안 나왔는지**는 눈에 보입니다

자동차의 수동 변속기와 같습니다. 기어를 사람이 넣지만, 엔진과 변속기의 메커니즘은 기계가 합니다.

이 시기가 이 이야기에서 가장 중요한 지점입니다. **완벽한 자동화가 아니라 "귀찮지만 작동하는 수동 파이프라인"**이 현실이었습니다. 영웅 서사가 아닙니다.

이 "자동화 시도 → 포기 → 수동 → 다시 시도" 패턴의 미시적 재현이 Husky(pre-commit hook)에서도 보입니다. 2월 15일에 설치했다가 같은 날 제거하고(`ec98fbe5`), 11일 뒤인 2월 26일(tsc 226→0 당일)에 다시 설치했습니다(`d6ba24b6`). 자동 게이트를 원하지만, 준비가 안 되면 오히려 방해가 됩니다.

## 3. 폭주 — LLM은 멈출 줄 모릅니다

수동으로 한 단계씩 누르면서도, 각 단계 안에서 문제가 터졌습니다. 가장 파괴적인 문제는 다음과 같습니다:

> **LLM은 "모르겠다"를 못 합니다.**

`/green`을 실행해서 구현을 시키면, 못 풀면 어떻게 할까요? 멈추는 게 아닙니다. **계속 수정합니다.**

```
수정 → 실패 → 다른 수정 → 실패 → 이전 수정 되돌림 → 또 수정 → ...
```

사람이 보고 있지 않으면, 돌아와보면 코드가 엉망이 되어 있습니다. 토큰만 소진되고, 원래 방향과 완전히 다른 곳에 가 있습니다.

여기서 중요한 깨달음이 왔습니다:

> **계획의 본질은 "좋은 설계"가 아니라 "멈출 수 있는 단위를 만드는 것"입니다.**

처음에 모두가 말했던 "충분히 계획을 세우세요"가 왜 중요한지, **이 폭주를 겪고 나서야** 체감했습니다.

Task가 크면:
- LLM이 중간에 길을 잃어도 "아직 끝 안 났으니까" 계속 갑니다
- 어디서 잘못됐는지 사후에도 찾기 어렵습니다
- 사람이 개입할 틈이 없습니다

Task가 잘게 쪼개져 있으면:
- 하나 끝 → **여기서 멈출 수 있습니다**
- 잘못됐으면 **이 Task만 버리면 됩니다**
- 다음 Task 시작 전에 **방향 점검이 가능합니다**

그래서 `/divide` 워크플로우가 나왔습니다. 목표에서 역추적하여 전제조건을 분해하고, 각 leaf가 "한 세션에 해결 가능한 크기"가 될 때까지 쪼갭니다:

```markdown
## /divide — Goal-Driven Backward Chaining

> **What**: 목표에서 역추적하여 미충족 전제조건을 재귀적으로 분해.
> **Output**: 모든 leaf가 ✅(충족) 또는 🔨(Work Package)인 전제조건 트리.
> **Constraint**: 코드 변경 없음. 모든 판정에 코드 증거.
```

`/divide`의 규칙 중 하나입니다:

```markdown
🔨 Work Package는 **한 /solve 세션에 해결 가능한 크기**여야 한다.
그보다 크면 더 분해한다.
```

이 "한 세션에 해결 가능한 크기"가 **브레이크 포인트**입니다. BOARD.md의 Now/Done이 프로젝트 관리가 아니라 **폭주 방지 장치**로 기능합니다.

내러티브가 원점으로 돌아왔습니다:

> "계획을 세우세요" (남의 말) → 안 함 → 폭주 → 쪼개야 멈출 수 있다 → **"계획을 세우세요" (내 경험)**

## 4. 복구 경로 — "다시 생각해봐" 시리즈

파이프라인이 일직선으로 흐르지 않는다는 걸 인정하자, **복구 경로**가 필요해졌습니다. 실전에서 벌어지는 상황들입니다:

| 상황 | 필요한 행동 |
|------|-----------|
| 막혔다, 뭔가 잘못됐는데 뭔지 모르겠다 | 멈추고 근본 원인을 찾아야 한다 |
| 방향이 맞는지 의심스럽다 | 의도와 결과를 비교해야 한다 |
| 확신이 있는데 틀렸을 수 있다 | 반대 관점에서 검토해야 한다 |
| 테스트가 실패하는데 왜인지 모르겠다 | 코드 수정 없이 원인만 분석해야 한다 |
| 답이 없다. 이 방향 자체가 틀렸다 | 포기하고 실패록을 남겨야 한다 |
| 품질 게이트를 통과 못 했다 | 이전 단계로 되돌려야 한다 |

전부 **"멈추고 생각해"**인데, 이걸 하나의 "다시 생각해봐"로 퉁치면 LLM은 뭘 해야 할지 모릅니다. 1편에서 배운 교훈을 적용합니다: **산출물이 다르면 워크플로우가 달라야 합니다.**

### `/why` — 근본 원인 진단

```markdown
## 트리거 조건
- 같은 작업을 3회 이상 반복하고 있다
- 10분 이상 진전 없이 머물러 있다
- regression 발생
- 결과물의 품질이 기대 이하다

## 1. 🛑 STOP — 완전 정지 (Andon)
코드를 더 고치지 않는다. 하던 것을 내려놓는다.
```

`/why`의 첫 번째 행동이 **"멈춰라"**입니다. 이게 핵심입니다. LLM은 스스로 멈추지 못하니까, 워크플로우가 강제로 멈추게 합니다.

산출물은 **WHY Report**입니다:

```markdown
## 🔴 WHY Report
**증상**: [한 줄 요약]
**Loop**: Single / Double
**근본 원인**: [5 Whys 결론] 또는 **전제 무효**: [무효화된 governing variable]
**컨텍스트 오류**: [Self-Audit에서 발견한 것]
**제안**: [올바른 방향 1~2개]
```

보고서를 제출한 후 **사용자가 방향을 결정할 때까지 절대 진행하지 않습니다**.

### `/reflect` — 경량 방향 점검

```markdown
## /reflect — 멈추고 돌아보기

> 보편 사이클의 어떤 단계 사이에든 삽입할 수 있는 cross-cutting 점검.

3가지 질문:
① 의도와 결과가 일치하는가?
② 건드리지 않아야 할 것을 건드리진 않았는가?
③ 다음 단계로 넘어가도 되는가?
```

`/why`가 무겁다면(Andon, 5 Whys, Double-loop), `/reflect`는 가볍습니다. 1분짜리 점검입니다. 산출물도 가볍습니다: "✅ reflect 통과" 한 줄이거나, "🔴 문제 있음" 보고입니다.

### `/redteam` — 외부 전문가 비판

자기가 만든 설계를 자기가 검증하면 편향이 생깁니다. `/redteam`은 LLM에게 "지금까지 한 것을 공격해봐"라는 역할을 줍니다. 산출물은 **Findings + Severity 분석표**입니다.

세 워크플로우의 차이를 표로 보면:

| | `/why` | `/reflect` | `/redteam` | `/diagnose` | `/ban` | `/rework` |
|---|--------|-----------|-----------|------------|--------|----------|
| **무게** | 무겁다 | 가볍다 (1분) | 중간 | 중간 | 극단적 | 가볍다 |
| **질문** | "왜 막혔나?" | "방금 한 거 맞아?" | "이거 틀렸을 수 있나?" | "테스트가 왜 깨지나?" | "이거 포기할까?" | "이전 단계로 돌아갈까?" |
| **산출물** | WHY Report | 한 줄 통과/실패 | Red Team 분석표 | 삽질 일지 | 실패록 + 인수인계 | 게이트 실패 → 루프백 |
| **시점** | 막혔을 때 | 매 단계 후 | 설계 완료 후 | 테스트 실패 시 | 답이 없을 때 | 게이트 탈락 시 |

### 복구 경로의 한계

솔직하게 말씀드리면, 이 복구 워크플로우들에는 **근본적 한계**가 있습니다. LLM이 자기가 헤매고 있다는 걸 모르니까, `/why`를 **스스로 호출하지 못합니다**. "멈춰"를 사람이 쳐야 하는데, 사람이 보고 있지 않으면 그냥 폭주합니다.

```
이상적:  plan → red → green → (이상 감지) → /why → 복구 → 계속
실제:    plan → red → green → (이상 못 감지) → 삽질 → 삽질 → 삽질 → 사람이 발견
```

자가 진단 불가 + 자가 중단 불가 = **외부 강제 장치 필요**. 이건 4편에서 `/auto`의 Stop Hook으로 이어집니다.

## 5. 프랙탈 — 같은 문제가 스케일을 바꿔 반복됩니다

이 편에서 반복적으로 나타난 패턴을 정리합니다:

```
/tdd → 한 턴에 Red+Green 우겨넣음     → /red + /green 분리로 해결
/go  → 한 세션에 모든 스킬 우겨넣음     → 수동으로 한 단계씩 눌러서 해결
/green → 못 풀면 멈추지 않고 폭주       → /why + /divide + Task 분해로 해결
```

**추상화 수준이 올라갈 때마다 같은 병이 재발하고, 같은 처방(분리 + 게이트)으로 치료됩니다.**

| 스케일 | 문제 | 해법 |
|--------|------|------|
| 턴 (micro) | 한 턴에 다 하려 한다 | 워크플로우 분리 |
| 세션 (meso) | 한 세션에 다 하려 한다 | 수동 게이트 (사람이 "다음" 누름) |
| 프로젝트 (macro) | 큰 Task를 한번에 하려 한다 | /divide + BOARD.md |

### 프랙탈의 가장 극적인 두 사례

**`/issue` — 하루에 4버전 (2026-02-21)**

이슈를 자율적으로 해결하는 `/issue` 워크플로우를 만들었습니다. `/go`의 마이크로 버전이라고 할 수 있습니다. 결과는 예상 가능합니다:

```
c3b04d96 refactor: /issue v2 — fully autonomous with embedded reflect/doubt/review
9ffd24a1 docs: discussion — /issue workflow safeguards (W1-W9)
4214551f refactor: /issue Phase 2 — recursive plan→reflect loop (max 3 rounds)
775be19e refactor: /issue v3 — dense, framework-referenced pipeline
```

하루 만에 4버전입니다. v1은 "다 알아서 해"였고 (→ `/go`와 같은 병), v2에서 reflect/doubt를 내장했지만 (→ 한 세션 함정), safeguards 논의를 거쳐 v3에서 "dense but bounded"로 안정됐습니다. **`/go`에서 겪은 것과 정확히 같은 시행착오가, 더 작은 스케일에서 하루 만에 재연**된 것입니다.

**Focus System — 파이프라인 이전의 반면교사 (2026-02-05~07)**

파이프라인이 없던 시절, Focus System은 3일 연속 재작성됐습니다:

```
ec997f4d 02-05 refactor(focus): rename pipeline to Sense→Intent→Update→Commit→Sync
c21e7fe6 02-06 fix: focus oscillation bug - remove aggressive focus recovery
b4a10c98 02-07 feat(focus): OS-Delegated Focus Recovery via pre-computed recoveryTargetId
```

02-05에 파이프라인을 리네이밍하고, 02-06에 oscillation 버그를 고치고, 02-07에 복구 전략을 완전히 바꿨습니다. `/why`도 `/reflect`도 없던 시절이라 **"왜 oscillation이 생기는가?"를 묻지 않고 증상만 고쳤고**, 결국 접근 자체를 갈아엎어야 했습니다. 복구 워크플로우가 있었다면 02-05에서 멈출 수 있었을 것입니다.

이 패턴을 알면, 새로운 스케일에서 같은 문제가 터졌을 때 **"아, 또 이거구나"**하고 바로 처방할 수 있습니다.

## 여기까지의 교훈

| 문제 | 발견 | 해법 |
|------|------|------|
| 수동 연결 피로 | `/go` — 상태 기반 라우터 | 판별표로 자동 라우팅 |
| 한 세션 함정 | `/go`가 다 우겨넣는다 | 수동으로 한 단계씩 (정직한 저점) |
| LLM은 못 멈춘다 | Dead Reckoning — 피드백 없이 폭주 | Task 분해 = 브레이크 포인트 |
| 복구가 필요하다 | "다시 생각해봐"도 산출물이 달라야 한다 | /why, /reflect, /redteam |
| 같은 문제 반복 | 프랙탈 — 스케일만 바뀐다 | 분리 + 게이트가 보편 처방 |

**핵심 한 줄**: 파이프라인은 직선이 아닙니다. **happy path + 복구 경로 + 브레이크 포인트**가 실전에서 동작하는 파이프라인입니다.
