# Working Process — 어떻게 일하는가

> Living Document. 작업 구조, 순서, 게이트에 대한 실천 지혜.

---

## Workflow는 진화한다 (Rigid → Adaptive)

첫 번째 workflow는 "최악의 경우"를 가정하고 모든 단계를 필수로 만든다. 실전에서 사용하면 불필요한 단계가 드러나고, 조건부 가드가 추가된다.

**적용**: 새 workflow 작성 후 3~5회 사용하면서 각 단계의 필요 조건을 명시하기. "항상 이것"에서 "조건 X이면 스킵"으로 진화.

---

## Workflow는 연결된다 (Isolated → Piped)

각 workflow가 독립적으로 끝나는 것에서, 다음 workflow를 트리거하는 형태로 진화. `/project` CLOSING → `/retrospect` 호출처럼.

**적용**: 프로젝트/세션 종료 시 회고를 자동 트리거. Workflow 간에 명시적 "다음 단계" 섹션 추가.

---

## 파일 이동은 Atomic

import 경로 변경과 파일 이동은 반드시 하나의 커밋 단위로. 동반 자산(CSS, SVG, JSON)도 함께 이동. 인프라 코드(vite-plugins, tsconfig)도 grep 범위에 포함.

**체크리스트**:
1. 대상 파일의 동반 자산 함께 이동
2. import 경로 변경은 파일 이동과 동시
3. 인프라 경로 검색 (vite.config, tsconfig paths)
4. 스모크 테스트

---

## 분류는 재확인한다

Known/Open/Constrained 판단이 틀릴 수 있다. `/divide` 재분류 단계가 필요. "이미 판단했으니까"로 진행하지 않기.

**적용**: 판단 후 조사 시작 전에 분류를 다시 확인. 특히 "blocked"는 측정 후에 결정.

---

## 엔트로피 감소 체크

변경 후 시스템의 고유 패턴 수가 늘었으면 해결이 아니라 악화다. 기존 메커니즘을 재사용하거나, 여러 특수 사례를 하나의 범용 메커니즘으로 대체해야 해결이다.

**체크**: "이 변경이 시스템에 새로운 유일한 패턴을 추가하는가?" 추가한다면 부채다.

---

## Think ≠ Do: Phase 전환 시 출력 형식도 전환

Discussion 단계에서 Toulmin 표를 반복 첨부했으나, 실행 단계에서는 불필요한 오버헤드. 논의(Think) 종료 후 실행(Do)에서는 작업 결과만 보고.

---

## 회고는 선택이 아니라 필수

Heavy 완료 후 회고를 건너뛰고 다음 프로젝트 진입 시 에러 진단 실패. Workflow 자가 개선이 다음 프로젝트의 효율을 좌우한다.
