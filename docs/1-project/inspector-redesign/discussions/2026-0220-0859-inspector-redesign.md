**🧑 사용자**: 내가 어떤 행동을 했는데 뭐가 안돼, 라고 했을 때 AI가 코드를 뒤적거려야만 현상을 파악할 수 있다는 게 비효율적이다. 내 상태를 통째로 넘겨주고 싶다.
**🤖 AI**: 그렇다면 런타임 상태를 복사해서 바로 넘길 수 있는 기능이 유용하겠군요. 구체적인 타겟은 무엇일까요?
**🧑 사용자**: 나도 볼 수 있어야 해. 내가 보는 건 단순 복사 버튼이 아니라 디버깅 정보여야 하지만, 노이즈가 되지 않는 선에서 제안해줘.
**🤖 AI**: Trigger(무엇을), Context(어디서), Mutation(무엇이 변했나 - Diff) 이 세 가지가 핵심 단서입니다. 이걸 시각적 타임라인으로 어떻게 보여드리면 좋을까요?
**🧑 사용자**: 이벤트 -> 커맨드 -> diff 형태가 타임라인으로 흐르고, 필요하면 전체 상태를 열어보는 구조가 좋더라. 그리고 문제를 만났을 때 이 묶음을 통째로 '확인해줘' 라고 할 수 있게. 지금 인스펙터는 노이즈가 너무 많아.
**🤖 AI**: 타임라인엔 채팅처럼 오토 스크롤이 필요하고, 상태 변화(Diff)가 없는 커맨드는 디버깅 가치가 없으니 숨기는 것이 어떨까요. 다만, Focus/Drag 같은 고빈도 이벤트는 어떻게 다룰까요?
**🧑 사용자**: OS 수준과 State 흐름을 분리하자. Focus는 OS 느낌이니까, 체크박스(Filtering)로 분리해서 껐다 켰다 하면 되겠지.

### Conclusion

**Current Intent**
전체 덤프나 소스 코드 파싱 대신, AI와 사용자 모두에게 노이즈 없는 핵심 인과관계(Trigger-Context-Diff)를 제공하여 디버깅 커뮤니케이션 비용을 줄인다.

**Warrants**
- W1. 현재 환경에서는 문제 발생 시점의 앱 전체 상태를 AI에게 전달하기 번거로워 정보 누락과 디버깅 핑퐁이 발생한다.
- W2. AI가 소스코드(정적)만 추적하게 두는 것은 런타임 인과관계 파악에 비효율적이다.
- W3. 전체 상태 덤프는 노이즈. 최소 단서 트리오(Trigger, Context, Diff)만 추출이 효율의 극치.
- W4. 타임라인은 `Event → Command → Diff` 구조이며, 전체 상태는 on-demand로 펼친다.
- W5. 상태 변경이 없는 빈 껍데기 커맨드는 부분적으로 숨겨 노이즈를 제어한다.
- W6. Focus 이동 등 고빈도 OS 레벨 이벤트는 State 변경과 채널을 분리하여 필터링 토글로 언제든 관찰/격리 가능토록 제어한다.

**한 줄 요약**:
인스펙터를 단순 상태 뷰어가 아닌, 시그널(Diff)과 노이즈(Focus/무변화)를 분리하고 필터링할 수 있는 **[타임라인 기반의 AI 디버깅 컨텍스트 캡처 도구]**로 재설계한다.
