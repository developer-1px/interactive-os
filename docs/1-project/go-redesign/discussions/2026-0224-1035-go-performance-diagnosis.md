# /go 성능 저하 진단 — Discussion

## Journey

**🧑 사용자**: go.md의 git 히스토리를 확인해달라. 원래 스텝별로 자세했는데 뭉개져있다.

**🤖 AI**: 3개 버전을 발견. v1(자율 루프 50줄) → v2(보편 사이클 134줄) → v3(Skip 제거, 148줄). v3에서 Skip 프리셋 삭제 시 스텝 상세 설명도 같이 사라짐. 22개 스텝 중 Step 16만 상세하고 나머지는 이름만 남음.

---

**🧑 사용자**: 생각 단계는 항상 /discussion → /project로 하고 있다. /go에는 필요없다.

**🤖 AI**: 실제 흐름(`/discussion → /project → /go`)과 go.md가 주장하는 흐름(`/go → Step 2: /discussion → ...`)이 완전히 다르다. go.md가 이미 끝난 생각 단계를 중복 강제하고 있다.

---

**🧑 사용자**: 만들기 단계가 더 중요하다. divide/blueprint/naming은 반드시 해야 하고, 특히 headless OS 방식으로 먼저 설계하고 검증을 끝내야 하는데, 이 단계가 없으니 그냥 React로 만들어버린다.

**🤖 AI**: 현재 "만들기" 섹션이 3줄(/tdd, /solve, /refactor)로 뭉개져 있다. LLM이 headless OS 설계를 건너뛰고 useState+onClick으로 회귀하는 근본 원인.

---

**🧑 사용자**: OS를 가지고 만드는 법 자체를 먼저 학습하거나 매뉴얼을 숙지해야 한다. 우리는 기능 구현이 아니라 OS가 제대로 돌아가는지 개밥먹기가 더 중요하다.

**🤖 AI**: LLM은 매 세션 새로 태어난다. OS 숙지 단계가 없으면 매번 React 패턴으로 회귀. 온보딩 절차가 필요하다.

---

**🧑 사용자**: 이 모든 프로젝트는 빌더의 완성이 아니라 OS의 완성이다.

**🤖 AI**: 프레임 전환 필요. "태스크를 해라" → "태스크를 통해 OS를 증명해라". 순수 React로 완성되면 OS를 안 썼다는 뜻이므로 실패.

---

**🧑 사용자**: OS 매뉴얼을 확인해달라.

**🤖 AI**: Kernel 문서 9개, SPEC.md 511줄, 벤치마크 앱(Todo), LLM 친화 설계 문서 존재. 하지만 **"앱 만드는 통합 런북"**과 **"headless OS로 설계하는 법"** 문서가 없다. 재료는 다 있는데 조리법이 없다.
