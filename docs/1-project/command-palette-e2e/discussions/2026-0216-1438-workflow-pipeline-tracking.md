# Discussion: 워크플로우 파이프라인 추적 + /perf 통합

## Journey

**🧑 사용자**: /perf를 /project나 /go에도 반영할 계획을 세워보자. 단계를 누락하는 경우도 해결하자.

**🤖 AI**: /perf를 /review에 성능 렌즈로 통합(예방) + 독립 워크플로우 유지(치료). 단계 누락은 체크리스트 출력, 게이트 필수화, 최소 루프+태그 중 선택.

---

**🧑 사용자**: 성능이 느리다는 건 어떻게 감지?

**🤖 AI**: ① 예방(코드 패턴 감지 in /review) ② 자동 측정(E2E 임계값) ③ 수동 체감. ①+③ 조합이 현실적.

---

**🧑 사용자**: 워크플로우 체크리스트 출력은 하되, 다 해야 하는 건 아님. 회고는 프로젝트 끝나면 해야 하는데 턴이 모자라면 안 되는 경우가 있어. 지금껏 자동으로 돌아간 적이 없어.

**🤖 AI**: 워크플로우 실행 상태가 세션 간 휘발되는 게 근본 원인. BOARD.md에 잔여 단계를 기록하면 다음 세션에서 발견 가능.

---

**🧑 사용자**: BOARD와 태스크별 workflow는 다른 개념. 추적 단위는 "태스크 × 파이프라인 단계"다.

**🤖 AI**: 맞다. B 방식(BOARD 태스크 하위 체크리스트)으로 태스크마다 파이프라인 진행 상태 기록.

---

**🧑 사용자**: B로 가서 되는지 확인하자. 미니멀하게.

> 핵심: 워크플로우 단계 누락은 "강제"가 아니라 "가시성 + 지속성"으로 해결한다.

---

## Conclusion

**Why**: 워크플로우 단계(특히 /retrospect)가 세션 경계에서 누락된다.

**Intent**: 세션이 끊겨도 "어디까지 했는지" 다음 세션에서 발견하여 이어갈 수 있게 한다.

**Warrants**:
- W1. `/perf` 핵심(구독 패턴 검증)은 `/review` 성능 렌즈로 예방 가능
- W2. `/perf`는 독립 워크플로우로 유지 (체감 이슈 시 수동 호출)
- W3. 추적 단위는 프로젝트가 아니라 **태스크 × 파이프라인 단계**
- W4. BOARD.md의 태스크 하위에 파이프라인 체크리스트를 두면 자연스러움
- W5. 과잉 절차가 아니라 기존 포맷에 서브아이템 추가일 뿐
- W6. `/go`가 BOARD를 읽을 때 잔여 단계를 발견하여 이어서 실행

**결정 사항**:
1. `/review`에 성능 패턴 검사 섹션 추가
2. `/project` BOARD 포맷에 파이프라인 체크리스트 예시 추가
3. `/go` 컨텍스트 파악에 "미완료 파이프라인 단계 확인" 추가
4. `/perf` 워크플로우 신규 생성 (이전 세션에서 완료)

> 워크플로우 누락은 "강제"가 아니라 "기록의 지속성"으로 해결한다.
