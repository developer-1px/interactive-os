# 파이프라인 회고: Story·Spec 품질 기준

## Journey

**🧑 사용자**: 파이프라인 검증 회고. UI까지 안 갔잖아? 프로젝트 레벨도 아니고.

**🤖 AI**: 맞다. resolveFieldValue 8줄은 프로젝트가 아니라 유틸 함수. 속도를 위해 가장 쉬운 태스크(T1 데이터 모델)만 돌렸다.

---

**🧑 사용자**: DT는 왜 안 했어?

**🤖 AI**: 데이터 모델 태스크라 "규칙상 스킵 가능"이라고 넘겼다. 검증 목적이면 인터랙션 태스크를 골랐어야 했다.

---

**🧑 사용자**: 우리는 FE야. 화면 중심으로 사용자 입력부터 끝까지 가는 기획이어야 해.

**🤖 AI**: /spec에서 수평 분해(모델→UI)가 아닌 수직 분해(입력→상태→화면)가 맞다.

---

**🧑 사용자**: 그전에 스토리부터 보강해야 해. UX까지 고려해서.

**🤖 AI**: Story AC가 "액션을 실행한다"로 추상적. UX Flow(화면 요소, 위치, 시각적 피드백)가 필요.

---

**🧑 사용자**: OS가 있잖아. 파이프라인 한 바퀴 돌 수 있는 기획이어야 해.

**🤖 AI**: Story의 최소 단위 = 입력→커맨드→상태→화면 1사이클. INVEST S를 시간이 아닌 파이프라인 사이클로 재정의.

---

**🧑 사용자**: DT를 못 만드는 기획이면 빠꾸여야 한다.

**🤖 AI**: DT Gate. DT가 나올 수 없는 Story = 입력이 정의 안 됨 = FE 기획 실패.

## Conclusion

| Toulmin | 내용 |
|---------|------|
| **🎯 Claim** | FE User Story의 품질 기준: ① DT를 쓸 수 있어야 한다 ② 파이프라인 1사이클(입력→커맨드→상태→화면)이 최소 단위다 ③ UX Flow가 포함되어야 한다 |
| **📊 Data** | builder-i18n 검증에서 DT 스킵, UI 미구현, 데이터 모델만으로 Green 통과 — 파이프라인 검증 불완전 |
| **🔗 Warrant** | DT 불가 = 사용자 입력 미정의 = FE 아님. 수직 분해해야 한 사이클 안에 UI가 포함됨 |
