## 1. 개요 (Overview)

Interactive OS의 "일하는 방식"은 세 계층으로 구성된다:

| 계층 | 파일 | 역할 | 규모 |
|------|------|------|------|
| **헌법** | `CLAUDE.md` + `.agent/rules.md` | 시점 독립 원칙, 매 세션 강제 노출 | ~170줄 × 2 |
| **워크플로우** | `.agent/workflows/*.md` | 절차적 강제 (Step = hard gate) | 41파일, ~3,000줄 |
| **실천 지혜** | `docs/2-area/praxis/` | 회고·discussion에서 축적된 패턴 | 7파일, ~400줄 |

추가로 `.agent/knowledge/*.md` (audit, spec 등 지식장부)와 `docs/2-area/references/` (46개 외부 기법 레퍼런스)가 보조한다.

**분석 대상**: 이 체계 전체가 자신이 선언한 원칙을 실제로 따르는가?

## 2. 🔴 Red Team 분석

### RT-01. 워크플로우 폭발 — 시스템이 자신의 원칙을 위반한다

**공격**: Goal #6 "학습 비용을 0으로 만든다", Project #8 "모든 산출물은 부채다"를 선언하면서, 41개 워크플로우 파일을 유지한다. 새 세션에서 에이전트가 `/go`를 실행하려면:

```
rules.md (170줄) → STATUS.md → BOARD.md → spec.md → 해당 workflow (50~160줄) → knowledge book → praxis 참조
```

최소 500~1,000줄의 컨텍스트를 로드해야 코드 1줄을 쓸 수 있다. **학습 비용이 0이 아니라, 매 세션마다 부트 비용이 발생한다.**

"같은 문제를 푸는 선택지가 여럿이면, 개체(concept)가 적은 쪽이 정답이다" (Project #1) — 워크플로우 시스템 자체가 이 규칙을 위반하고 있지 않은가?

### RT-04. 메타-작업 오버헤드 — 과정이 목적을 삼킨다

**공격**: 하나의 기능을 구현하려면:

```
/plan → /spec → /red → /green → /bind → /audit → /verify → /refactor → /retrospect → /archive
```

**최소 10단계**. 각 단계마다 산출물(문서, 표, 체크리스트)을 생산해야 한다. Project #8 "모든 산출물은 부채다"와 정면으로 충돌한다.

Light 프로젝트에서도 `/spec` (BDD + DT) → `/red` (실패 테스트) → `/green` (구현) → `/audit` (위반 검사) → `/verify` (5-gate)는 **의무**다. 3줄짜리 버그 수정에 500줄의 프로세스 문서가 붙는 상황이 가능하다.

---

### RT-05. 관찰자 효과 — 산출물이 의식(ritual)이 된다

**공격**: Working #13 "산출물(Noun)로 증명한다"는 LLM의 goal fixation을 방지하기 위해 도입됐다. 그런데 산출물 자체가 goal fixation의 대상이 될 수 있다.

LLM이 Cynefin 분류표를 "채워야 하니까" 작성하면, 분석 없는 분류표가 된다. 체크리스트를 "통과해야 하니까" 체크하면, 검증 없는 통과가 된다. **산출물의 존재가 사고의 증거가 되지 않는다.**

시스템 자체가 이를 인식하고 있다 (llm-behavior.md "Goal Fixation으로 절차를 건너뛴다") — 하지만 교정법이 "규칙에 근거를 함께 기록"뿐이다. 근거를 읽어도 goal fixation하는 LLM에게는 근거도 형식적 텍스트가 된다.

---

### RT-06. Cynefin 만능 망치 — 사소한 결정에도 프레임워크

**공격**: Cynefin은 모든 판단 분기에서 사용된다: `/divide`, `/go`, `/inbox`, `/plan`, `/solve`. import 누락 수정도 "Clear → Sense → Categorize → Respond"로 분류된다.

Cynefin은 **조직적 의사결정의 sensemaking framework**이지, 코드 수준 판단 도구가 아니다 (Dave Snowden 본인도 micro-level 적용에 회의적). 모든 것에 적용하면 trivial한 결정이 모두 "Clear"로 분류되고, 프레임워크가 형식적 체크박스로 전락한다.

---

### RT-07. "No DOM" 절대주의의 비용

**공격**: "OS 코드는 DOM을 직접 조작하지 않는다" — 3-commands뿐 아니라 **OS 전체**에서 금지.

이 OS는 **웹 브라우저 위에서 동작한다**. 결국 누군가는 DOM을 건드려야 한다. 추상화 계층을 강제하면:
- 간단한 `.focus()` 호출이 `ctx.inject()` → kernel state → command pipeline → 최종 DOM 조작으로 5단계를 거친다
- OS gap이 발견될 때마다 "기존 기능이 없다 → OS 기능 추가"로 OS의 표면적이 계속 커진다
- Project #1 "시스템의 고유 패턴 수가 늘었으면 해결이 아니라 악화다" — OS 자체의 패턴 수 증가는 엔트로피 증가 아닌가?

## 3. 🔵 Blue Team 방어

### BT-01. 워크플로우 폭발 방어

**방어**: `/go`는 **라우터**다. 41개를 전부 읽는 게 아니라, 상태에 따라 **1개만** 로드한다. "Red 테스트 없음" → `/red`만 읽는다. 나머지 40개는 존재하지 않는 것과 같다.

부트 비용은 `rules.md` + `STATUS.md` + `BOARD.md` ≈ 300줄이고, 이것은 **프로젝트를 이해하는 최소 비용**이다. 이 비용 없이 코딩을 시작하면 "잘못된 방향으로 100줄" (Working #4)이 더 비싸다.

41개가 많아 보이지만, 각 워크플로우는 SRP(단일 책임)를 따른다. 하나를 합치면 다른 곳에서 불필요한 절차를 건너뛰기 어려워진다.

### BT-02. 규칙 삼중 복제 방어

**방어**: 세 위치의 역할이 다르다:

| 위치 | 역할 | 수명 |
|------|------|------|
| `CLAUDE.md` | 강제 노출, 매 세션 읽힘 | 헌법 (거의 변경 안 함) |
| `rules.md` | 명시적 로드용 참조 | = CLAUDE.md 미러 (동기화 의무) |
| `praxis/` | **사례 + 교훈** 기록 | 계속 축적 (덮어쓰기) |

`praxis/working-process.md`의 "엔트로피 감소 체크"는 규칙의 **복제**가 아니라, 규칙이 생긴 **사례와 체크리스트**다. 규칙(원칙)과 사례(적용법)는 다른 문서다.

CLAUDE.md ↔ rules.md 동기화는 인정한 부채다. 이상적으로는 하나여야 한다. 그러나 현재는 CLAUDE.md가 Claude Code의 강제 노출 경로이고, rules.md가 워크플로우의 명시적 로드 경로라는 기술적 이유로 분리되어 있다.

### BT-03. 프로젝트 급증 방어

**방어**: "Active"는 "동시 작업 중"이 아니다.

- 🔥 Focus = **지금** 작업 중 (2개)
- 🟢 Active = 시작했고, 재개 가능 (10개)
- 🟡 Paused = 의식적으로 멈춤 (2개)

Working #3 "하나를 끝내고 다음으로"는 🔥 Focus 메커니즘으로 강제된다. Focus가 2개 초과하는 것은 규칙 위반이다. Active 프로젝트는 각각 상태가 저장되어 있으므로 context switch 비용이 **0에 가깝다** — BOARD.md를 읽으면 마지막 상태에서 재개 가능하다.

### BT-04. 메타-작업 오버헤드 방어

**방어**: `/plan`의 규모 판정이 이 문제를 다룬다:

| 규모 | 의무 단계 | 스킵 가능 |
|------|----------|----------|
| Heavy | spec → red → green → bind → audit → verify → refactor | 없음 |
| Light | spec → red → green → audit → verify | refactor, prd |
| Meta | 태스크 직접 실행 | red/green 스킵 |

3줄짜리 버그 수정은 `/issue`로 라우팅되어 별도 경로를 탄다. `/go` #1.5 "Meta 프로젝트"는 Red/Green 자체를 스킵한다. 10단계는 Heavy의 최대 경로이지, 모든 작업의 필수 경로가 아니다.

또한, **"과정이 목적을 삼킨다"는 앱 개발의 관점**이다. 이 시스템에서 "모든 프로젝트의 목적은 앱의 완성이 아니라 OS의 완성이다." OS 개선 지식(official, praxis, rules)은 메타-작업의 부산물이 아니라 **주 산출물**이다.

### BT-05. 관찰자 효과 방어

**방어**: "의식적 산출물이 사고를 보장하지 않는다"는 사실이다. 그러나 **산출물 없는 사고는 검증 불가능하다**.

두 선택지:
1. 산출물 강제 → 일부는 형식적이지만, **틀린 산출물은 발견 가능**
2. 산출물 없음 → LLM이 "분석했습니다"라고 말하면 **반박 불가능**

형식적 산출물이라도 사용자가 "이 Cynefin 분류가 틀렸다"고 교정할 수 있다. 머릿속 분석은 교정 기회 자체가 없다. **교정 가능성 > 완벽한 분석**.

### BT-06. Cynefin 만능 망치 방어

**방어**: 이 시스템에서 Cynefin은 **전체 분석 프레임워크가 아니라 triage gate**다.

```
Clear → 바로 실행 (분석 비용 0)
Complicated → 분석 후 실행
Complex → 나누거나 묻는다
```

"import 누락 = Clear"는 **2초짜리 판정**이다. Cynefin이 무거운 게 아니라, Clear 판정이 빠른 탈출구를 제공한다. Cynefin이 없으면 에이전트가 모든 작업에 동일한 무게의 프로세스를 적용한다 — 그것이 더 비싸다.

### BT-07. "No DOM" 절대주의 방어

**방어**: **이것이 OS의 존재 이유다.**

"브라우저에 위임하면 둘 다 잃는다. 브라우저에 떨어지는 입력은 OS의 실패다." `.focus()` 하나를 허용하면 **모든 곳에서** `.focus()`를 쓰는 슬리퍼리 슬로프가 시작된다.

OS gap 발견 → OS 기능 추가는 **의도된 성장 메커니즘**이다. OS의 표면적이 커지는 것은 엔트로피 증가가 아니라, **앱이 필요로 하는 추상화가 늘어나는 것**이다. 각 추상화는 여러 앱에서 재사용되므로 시스템 전체의 패턴 수는 줄어든다.

### BT-08. 지식 7중 분산 방어

**방어**: 7곳이 아니라 **4개의 생명주기(lifecycle)**다:

| Lifecycle | 위치 | 갱신 빈도 |
|-----------|------|----------|
| 헌법 (안 변함) | CLAUDE.md + rules.md | 연 2~3회 |
| 절차 (진화함) | workflows/ + knowledge/ | 프로젝트마다 |
| 지혜 (축적됨) | praxis/ | 회고마다 |
| 참조 (불변) | references/, official/ | 거의 변경 없음 |

같은 원칙이 여러 곳에 나타나는 것은 **cross-referencing**이지 duplication이 아니다. CLAUDE.md에 "엔트로피 감소"가 있고, praxis에 "엔트로피 감소 체크리스트"가 있는 것은 원칙과 적용법의 분리다.

### BT-09. 순환 루프 방어

**방어**: `/audit` → 루프백은 **품질이 올라가는 방향**의 루프다. 무한 루프가 되려면 매번 새로운 🔴가 발견되어야 하는데, 유한한 코드에서 🔴가 무한히 발생할 수 없다.

실무적으로: `/audit` → `/bind` 루프백 → 재감사에서 같은 파일의 같은 패턴을 수정하므로, 2~3회 이내에 수렴한다. AUDITBOOK(§2)에 알려진 OS 갭이 기록되므로, 같은 🟡를 반복 발견하지 않는다.

인정할 점: 명시적 종료 조건(최대 N회)을 `/go` ↔ `/audit` 루프에 추가하는 것은 개선 포인트다.

### BT-10. 검증 이상과 현실 방어

**방어**: Active 프로젝트가 증빙 없는 것은 **아직 완료 단계에 도달하지 않았기 때문**이다. Red→Green 증빙은 **완료 시** 붙는다.

Completed 프로젝트 12개 **전부**에 증빙이 있다:
- `sense-purity`: "+13 tests, audit+doubt+retrospect 완료"
- `normalized-collection`: "109 tests GREEN"
- `unified-pointer-listener`: "+13 tests, 456줄 삭제"

방법론의 ROI는 **완주된 프로젝트에서만** 측정 가능하다. 미완주 프로젝트는 ROI의 분자(산출)에도 분모(투입)에도 아직 포함되지 않았다.

### BT-11. /self 부재 방어

**방어**: 인정. Discussion에서 결정됐지만 구현되지 않았다. 이것은 **backlog**이지 결함은 아니다. 결정 시점에 더 높은 우선순위 작업(builder-v2, dropdown-dismiss)이 있었다.

다만, discussion-expert-upgrade.md가 1-project가 아닌 2-area/praxis에 있으므로 프로젝트로 전환되지 않았다. `/project`로 스캐폴딩하고 BOARD에 등록해야 추적 가능하다.

### BT-12. 46개 기법 방어

**방어**: 46개 기법이 모두 **실제 워크플로우에서 참조**된다. 레퍼런스 문서의 "참조" 섹션이 이를 증명한다:

- Cynefin → `/divide`, `/go`, `/inbox`, `/plan`, `/solve`
- Toulmin → `/discussion`
- 5 Whys → `/issue`, `/solve`
- Lean 7 Muda → `/doubt`
- KPT → `/retrospect`

"알고 있다 vs 사용한다" — 워크플로우가 기법명을 참조하는 것은 **사용 의도의 선언**이다. 레퍼런스 문서는 그 의도가 정확히 실행되도록 정의를 제공한다.

부채 여부: 레퍼런스 문서 7개는 **불변 참조**다. 갱신할 필요가 거의 없으므로 유지 비용이 0에 가깝다.

## 5. 결론 (Conclusion) — 개선 제안

### 즉시 실행 가능 (🔴 인정 항목)

1. **RT-02**: `CLAUDE.md` ↔ `rules.md` 통합 방안 조사. 기술적으로 하나의 파일로 만들 수 있는지 확인. 불가능하면 rules.md를 "CLAUDE.md를 읽어라"는 포인터로 축소.
2. **RT-09**: `/go` 워크플로우에 `/audit` 루프백 최대 횟수(3회) 명시. 3회 후에도 🔴가 남으면 사용자에게 보고하고 멈춘다.
3. **RT-11**: `/self` 워크플로우를 프로젝트로 등록. discussion-expert-upgrade.md의 결론을 입력으로 사용.

### 중기 개선 (🟡 부분 인정 항목)

4. **RT-01**: `/go` 부트 시퀀스 최적화. rules.md 전체가 아닌 현재 Phase에 필요한 섹션만 로드하는 방식 고려.
5. **RT-04**: Light 프로젝트의 `/audit` 간소화. 위반 0건이 3회 연속이면 Light에서 /audit을 선택적으로 만드는 규칙 추가.
6. **RT-05**: `/audit` Step 1 결과에 "이 산출물이 형식적인가?" 자가 체크 1항 추가.
7. **RT-08**: praxis 문서에서 rules.md 원칙을 **인용**(링크)하되 **복제**하지 않는 규칙 명시.
