# /go 파이프라인 개선: 정량지표 검증 턴 도입

> 작성일: 2026-03-09
> 태그: infra
> 우선순위: P1

## 문제 / 동기

`/go` 파이프라인이 각 단계(`/spec`, `/red`, `/green` 등)를 **1턴=1샷**으로 실행한다.
한 턴에 완벽하게 안 나오는 작업에서 누락이 발생하고, 그 누락이 뒤 단계에서 터진다.

근본 원인 3가지:
1. **One-shot 한계**: 한번에 안 나올 수 있는 걸 한번에 하려고 하니까 누락
2. **자기 검증의 한계**: 같은 턴 안에서 자기 산출물을 객관적으로 못 봄 (코드 리뷰와 같은 원리 — 작성자 ≠ 검증자)
3. **의미적 자기 평가 신뢰 불가**: LLM이 "다 했어요"라고 판단하면 통과 — 정량 기준 없음

발견 경위: tsc/lint 절대 0 작업에서 `/green`이 한 턴에 완료를 선언했지만 에러가 남아있었고, 이후 수동으로 여러 턴 반복해야 실제 0에 도달.

## 현재 상태

- `/go`는 **순수 라우터** — 판별 → 워크플로우 호출 → 재진입 → 다음 단계
- 각 워크플로우는 내부에 "verify 실패 시 돌아간다" 한 줄이 있지만, 실제로 **여러 턴 반복하라는 명시 없음**
- 기계적 검증(tsc, lint, test PASS)은 있지만 `/go` 레벨에서 독립적으로 재확인하지 않음
- 의미적 완성도(spec 분량, 구현 커버리지)는 LLM 자기 판단에 의존

관련 파일:
- `.claude/skills/go/SKILL.md` (slash command로 정의됨)
- `.agent/workflows/verify.md`
- `.agent/workflows/green.md`, `red.md`, `spec.md`

## 기대 상태

`/go`를 **라우터 + 검증자**로 승격:

1. 각 단계 완료 후 **독립된 검증 턴**을 갖는다
2. 검증 기준 = **정량지표 게이트** (LLM 주관 판단 아닌 숫자 대조)
3. 미달이면 같은 단계에 **다시 턴을 할당** (재시도)
4. 통과하면 다음 단계로 라우팅

### 정량지표 후보

| 단계 | 지표 | 측정 가능성 |
|------|------|------------|
| `/spec` | DT 행 수 ≥ 기대치 | **미해결** — 기대치를 누가 정하나 |
| `/red` | DT 행 수 = `it()` 수 | 기계적 (파일 파싱) |
| `/green` | DT 행 = PASS `it()` + tsc 0 + lint 0 | 기계적 (vitest + tsc + biome) |
| `/audit` | grep 패턴 수 x 대상 파일 수 = 전수 실행 증빙 | 기계적 (실행 로그) |

### 완료 조건 (Done Criteria)

- [ ] `/go`에 단계 완료 후 검증 턴 로직이 추가됨
- [ ] 검증 기준이 정량 지표로 정의됨 (주관 판단 아님)
- [ ] 미달 시 재시도 + 재시도 한도(무한루프 방지) 구현
- [ ] 기존 파이프라인(`/auto` 포함)과 호환

### 검증 방법

- `/go`로 Light 이상 프로젝트 1개를 돌려서, 검증 턴이 누락을 실제로 잡는지 확인
- 재시도 한도에 도달하는 케이스에서 사람에게 보고하고 멈추는지 확인

## 미해결 Gap

1. **`/spec` 분량의 정량 기준**: DT 행이 충분한가를 LLM이 판단하면 같은 문제 회귀. 외부 기준(Zone 수 x 입력 종류 = 최소 DT 행) 같은 공식이 필요할 수 있음
2. **재시도 한도 설계**: 몇 턴까지 허용? 3턴? 한도 도달 시 행동은? (멈추고 보고 vs 다음 단계 강제 진행)
3. **자유형과의 균형**: 턴이 늘어도 가드레일(구조)이 있어야 퀄리티 유지. "알아서 해"는 퀄리티 폭락

## 접근 방향

1. `/go` 파이프라인 표에 각 단계 뒤 **검증 행**을 추가
2. 검증 행의 판별 로직 = 정량지표 체크 스크립트 (bash)
3. 미달 → 같은 단계 번호로 재라우팅 (재시도 카운터 관리)
4. 한도 초과 → `⛔ 보고하고 멈춘다`
