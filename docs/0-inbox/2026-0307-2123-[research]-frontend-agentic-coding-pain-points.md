## 1. 개요 (Overview)

프론트엔드에서 AI 에이전트가 자율적으로 코드를 작성·수정하는 "에이전틱 코딩"은 빠르게 확산 중이나, 백엔드와 구분되는 프론트엔드 고유의 구조적 한계에 부딪히고 있다. 가장 널리 알려진 "AI는 눈이 없다(시각적 결과를 검증할 수 없다)"를 넘어, **최소 6가지의 독립적 페인포인트**가 업계에서 반복적으로 보고되고 있다.

## 2. 분석 (Analysis)

### 👑 De facto Standard — 업계가 공통적으로 인정하는 핵심 페인포인트

| # | 페인포인트 | 설명 | 출처 |
|---|-----------|------|------|
| **P1** | **시각 검증 불가 (Blindness)** | AI는 렌더링 결과를 볼 수 없어 레이아웃 깨짐, 색상 부조화, 반응형 오류를 감지하지 못한다. 자동화된 접근성 테스트도 전체 이슈의 ~30%만 잡아낸다. | [Medium](https://medium.com), [LogRocket](https://logrocket.com) |
| **P2** | **접근성(a11y) 누락** | AI가 생성한 UI 코드는 시맨틱 HTML 대신 `<div>`/`<span>`을 남용하고, ARIA 속성을 누락하며, 포커스 관리를 무시한다. 결과적으로 키보드 전용 사용자와 스크린 리더 사용자를 배제하는 코드가 나온다. | [LogRocket](https://logrocket.com), [Medium](https://medium.com) |
| **P3** | **키보드 인터랙션 무시** | ESC로 모달 닫기, Tab 순서 관리, 포커스 트랩 등 키보드 기반 상호작용이 거의 항상 누락된다. AI는 "클릭으로 동작하는 UI"는 잘 만들지만, "키보드로 조작 가능한 UI"는 못 만든다. | [LogRocket](https://logrocket.com), [Medium](https://medium.com) |
| **P4** | **컨텍스트 윈도우 한계 → 아키텍처 표류** | 프론트엔드는 파일이 수십~수백 개로 분산(컴포넌트, 스타일, 상태, 라우팅)되어 있어 AI의 컨텍스트 윈도우를 쉽게 초과한다. 결과적으로 동일 유틸리티를 여러 파일에 중복 생성하고, 기존 아키텍처 패턴을 무시한 코드를 생성해 **아키텍처 표류(Architectural Drift)**를 유발한다. | [Medium](https://medium.com), [TowardsAI](https://towardsai.net) |
| **P5** | **상태 관리의 복잡성** | 전역 상태(Redux, Zustand)와 로컬 상태, 서버 상태(React Query)가 혼재하는 프론트엔드 특유의 상태 구조를 AI가 정확히 추적하지 못한다. Race condition, 무한 리렌더링, useEffect 남용 등의 버그를 양산한다. | [Talent500](https://talent500.com), [TheNewStack](https://thenewstack.io) |
| **P6** | **디자인 의도 상실 (Brand Sameness)** | AI가 생성한 UI는 기능적으로 동작하지만, 브랜드 아이덴티티나 디자인 시스템의 미묘한 뉘앙스를 반영하지 못한다. AI 의존도가 높을수록 제네릭한 "AI 느낌" 레이아웃이 나온다. | [AdvertisingBusiness](https://advertisingbusiness.org), [TheNewStack](https://thenewstack.io) |

### 🔄 Alternatives — 이 문제를 해결하려는 최신 시도들

| 접근법 | 설명 | Pros | Cons |
|--------|------|------|------|
| **Snapshot 기반 시각 검증** (Vercel agent-browser 등) | DOM을 Accessibility Tree로 압축하여 AI에게 제공, 토큰 93% 절감 | 기존 웹사이트에 범용 적용 가능 | DOM이 존재해야 함. 브라우저 실행 비용 발생 |
| **Design Token 강제 주입** | AI에게 디자인 시스템의 토큰(색상, 간격, 타이포)을 프롬프트/컨텍스트로 강제 주입 | Brand Sameness 완화 | 토큰이 컨텍스트 윈도우를 잡아먹음 |
| **상태 로직과 뷰의 분리** (Interactive OS 방식) | AI가 순수 데이터 모델과 커맨드만 작성하고, 인터랙션은 OS 레이어가 보장 | 접근성·키보드·포커스가 자동 보장. 토큰 효율 극대화 | 자체 OS 위 앱에만 적용 가능. 범용 웹에는 미적용 |
| **E2E 시각 회귀 테스트** (Playwright + 스크린샷 비교) | AI가 수정 전후 스크린샷을 비교하여 시각적 변화를 감지 | "눈" 문제를 부분적으로 해결 | Flaky 테스트 발생, 느린 실행 속도, 미세한 차이 감지 어려움 |

### 🚫 Anti-patterns — 프론트엔드 에이전틱 코딩에서 피해야 할 것들

| 안티패턴 | 왜 위험한가 |
|----------|------------|
| **"전체 UI를 한 번에 생성"** | 스타일·로직·마크업·접근성을 한꺼번에 맡기면 모든 축에서 품질이 떨어진다. 관심사를 분리하여 단계별로 생성해야 한다. |
| **순환 수정 (Circular Fixes)** | 렌더링 에러에 대해 AI가 `useEffect`를 추가하고, 그로 인한 무한 루프를 `useRef`로 막고, 그로 인한 상태 꼬임을 또다른 Effect로 막는 악순환. 근본 원인을 모른 채 증상만 때운다. |
| **접근성 후처리** | "일단 동작하게 만들고 나중에 ARIA를 붙이자"는 접근. 시맨틱 구조가 틀어진 상태에서 ARIA를 입히면 오히려 스크린 리더에 혼란을 준다. |
| **자기 검증 (Self-Verification)** | AI가 자기 코드를 자기가 테스트하는 것. Pulumi가 "Ralph Wiggum Loop"라 비판한 패턴. False positive가 발생하여 버그를 놓친다. |

## 3. 결론 및 제안 (Conclusion & Proposal)

### 핵심 발견

프론트엔드 에이전틱 코딩의 페인포인트는 **6가지 독립적 축**으로 구성된다:

```
P1. 시각 검증 불가 (Blindness)        ← 사용자가 이미 인식
P2. 접근성(a11y) 누락                 ← OS가 해결
P3. 키보드 인터랙션 무시               ← OS가 해결
P4. 컨텍스트 윈도우 한계 → 아키텍처 표류 ← OS가 완화 (13개 커맨드로 응집)
P5. 상태 관리의 복잡성                 ← OS가 완화 (NormalizedCollection + Pipeline)
P6. 디자인 의도 상실                   ← OS 범위 밖 (디자인 시스템 영역)
```

### 제안

Interactive OS는 P2, P3을 **완전히 해결**하고, P4, P5를 **구조적으로 완화**하는 위치에 있다. 이것은 6개 축 중 4개를 커버한다. P1(시각 검증)은 Vercel agent-browser 등의 외부 도구와 보완적이며, P6(디자인 의도)은 디자인 시스템(Teo)이 담당하는 영역이다.

**"AI에게 프론트엔드를 맡기면 왜 장난감만 나오는가?"에 대한 답은 P2~P5가 해결되지 않았기 때문이며, Interactive OS는 이 4개 축을 OS 레이어에서 보장하는 현재 유일한 프레임워크다** — 이 포지셔닝을 프로젝트 문서(README, 발표 자료 등)에 명시하는 것을 추천한다.
