# Discussion Journey — PARA 프로젝트 관리

> **날짜**: 2026-02-12
> **주제**: PARA에서 Project 관리에 대하여

---

**🧑 사용자**: 우리 PARA에서 project 관리에 대하여 (논의해보자)

**🤖 AI**: 현재 `1-project/os-core-refactoring`에 20개 문서가 플랫하게 나열. "문서 덤프"가 되고 있는 문제를 의식하고 계신 것 같다. 어떤 불편함을 느끼시는지?

---

**🧑 사용자**: 기간보다는 **목표 기반**. 대부분 프로젝트 관련 내용이 모이는데, **한 장의 A-Z 현황판** + 나머지는 자료 수준. **프레임워크가 있으면 좋겠다**.

**🤖 AI**: 두 가지로 나눠서 생각할 수 있겠다 — 거의 변하지 않을 **폴더 구조** + 표준화된 **문서 포맷**.

**🧑 사용자**: 맞아 2가지야. **폴더구조**, 그리고 **표준화된 문서 포맷**.

---

**🧑 사용자**: 쓰레기통까진 아니지만 잡다한 내용 모아두는 곳 + **정제된 몇 가지**만 있으면 될 듯.

**🤖 AI**: → 프로젝트 내부는 **2계층**: 정제된 문서(루트) + 원자료(서브폴더). PARA 자체의 축소판!

---

**🧑 사용자**: **기획서, 제안서, 현황판, 회고** — 이건 필수. ADR과 discussion은 옵셔널. **표준화된 관리도구는 없어?**

**🤖 AI**: 마크다운 기반 통합 프로젝트 문서 표준은 업계에 없다. ADR(MADR), RFC 등 조각은 있지만 통합은 자체 구축 필요.

---

**🧑 사용자**: **기획서는 What, 제안서는 How**. Why, What, How로 생각.

**🤖 AI**: PMBOK 4개 프로세스 그룹과 1:1 매핑되는 구조. 기획서 안에 Why→What 통합이 관리 비용 측면에서 유리.

---

**🧑 사용자**: 제안서는 하다보면 쌓이고 많이 변화하지. **notes에 넣고 채택된 게 올라가는 구조**로?

**🤖 AI**: 승격 메커니즘 — Draft(notes) → Adopted(루트) → Archived. PARA의 재귀적 축소판!

---

**🧑 사용자**: 대기업에서 이미 검증된 프로세스도 추천해봐. **선진문물대로 한번 해보고 싶어**.

**🤖 AI**: Google Design Doc, Amazon PR/FAQ, Basecamp Shape Up 비교. **Shape Up이 솔로 개발자에 가장 적합** (시간 고정/범위 가변). Shape Up + Google Design Doc 하이브리드 추천.

---

**🧑 사용자**: 기획서는 잘 안 바뀌고 제안서는 계속 바뀌니까 **분리 유지**. 기획서 → 제안서 → 현황판(Task 포함) → 회고 순으로.

**🤖 AI**: **변동성이 다른 문서를 합치면 안 된다** — Shape Up Pitch 통합은 고정 사이클에서만 유효. 장기 프로젝트에서는 분리가 맞다.

---

**🧑 사용자**: 그러면 프로젝트 폴더 하나 정해서 **새 구조로 리팩토링해봐**. 내가 보고 판단할게.

**🤖 AI**: `os-core-refactoring`(20개 문서)을 선택하여 실행 → 완료.

```
docs/1-project/os-core-refactoring/
├── SPEC.md       ← 기획서 (Why+What)
├── PROPOSAL.md   ← 제안서 (How, 채택된 전략)
├── STATUS.md     ← 현황판 + Tasks
├── RETRO.md      ← 회고
└── notes/        ← 기존 20개 문서 원본 보존
```

---

**한 줄 요약**: 프로젝트 문서 관리의 본질은 **"변동성에 따른 계층 분리"** — 불변(기획) / 가변(제안) / 상시(현황) / 사후(회고)를 나누고, 원자료에서 정제 문서로의 **승격 흐름**을 만드는 것이다.
