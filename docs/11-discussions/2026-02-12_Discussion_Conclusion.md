# Discussion Conclusion — PARA 프로젝트 문서 프레임워크

> **날짜**: 2026-02-12
> **주제**: PARA에서 Project 관리 방법론

---

## Why (동기)

현재 `docs/1-project/` 아래 프로젝트 폴더들에 문서를 체계 없이 축적하고 있다. 프로젝트 간 구조가 제각각이고, 현황 파악이 어렵고, 문서 라이프사이클 관리가 안 되는 문제를 해결하고자 한다.

## Intent (의도)

프로젝트 폴더를 **"구조화된 프로젝트 허브"** 로 만드는 표준 프레임워크를 구축한다. 완벽보다는 **"현재보다 나은 v1"** 을 만들어 이터레이션의 기반으로 삼는다.

## Warrants (논거 구조)

### 프레임워크의 두 축
- **W7**: 폴더 구조(where) + 문서 포맷(what)으로 분리
- **W8**: 폴더 구조는 프로젝트 성격에 무관하게 불변

### 2계층 구조
- **W9**: 프로젝트 내부 = 정제층(루트) + 원자료층(notes/)
- **W10/W19**: PARA의 재귀적 축소판 — notes ≈ inbox, 루트 ≈ project

### 필수 4종 문서
- **W15**: Why → What → How 논리 축
- **W28**: 변동성이 다른 문서는 분리해야 한다 (기획서 🟢 vs 제안서 🔴)
- **W31**: SPEC → PROPOSAL → STATUS → RETRO 순서 확정

| 문서 | 핵심 질문 | 변동성 | 시점 |
|------|----------|--------|------|
| **SPEC** (기획서) | Why + What | 🟢 낮음 | 프로젝트 초기 |
| **PROPOSAL** (제안서) | How | 🔴 높음 | notes에서 승격 |
| **STATUS** (현황판) | Where + Tasks | 🟡 상시 | 지속 갱신 |
| **RETRO** (회고) | So What | 🟢 낮음 | 마일스톤 시 |

### 검증
- **W21**: 4종 문서가 PMBOK 4개 프로세스 그룹에 1:1 매핑
- **W22**: 대기업 결재 체계(기안→품의→보고)와도 동일 구조
- **W24~W27**: Shape Up(솔로 적합) + Google Design Doc(기술 문서) 하이브리드 참조

### 승격 메커니즘
- **W18/W20**: 제안서는 notes에서 시작 → 채택 시 루트로 승격 (Draft → Adopted → Archived)

## 실행 결과

`os-core-refactoring` 프로젝트에 프레임워크 적용 완료:
```
docs/1-project/os-core-refactoring/
├── SPEC.md, PROPOSAL.md, STATUS.md, RETRO.md  (정제 문서 4종)
└── notes/  (기존 20개 문서 원본 보존)
```

---

**한 줄 요약**: 프로젝트 문서는 **변동성이 다른 4종(기획/제안/현황/회고)**으로 분리하고, **2계층(정제+원자료)**으로 쌓되, **승격 메커니즘**으로 라이프사이클을 관리한다.
