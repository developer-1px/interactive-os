# 목표: 인터랙션 OS (Interaction OS) 철학

## 1. 핵심 문제 (The Core Problem)
현대 웹 개발은 **"핸들러 지옥 (Handler Hell)"**으로 고통받고 있습니다.
비즈니스 로직은 수천 개의 `onClick`, `onChange`, `useEffect` 훅에 흩어져 있으며, 뷰(View) 계층과 깊게 결합되어 있습니다. 이로 인해 애플리케이션은 테스트하기 어렵고, 디버깅이 힘들며, 리팩토링하기에 취약해집니다.

## 2. 해결책: 인터랙션 OS (Interaction OS)
우리는 웹 애플리케이션을 단순한 페이지들의 집합이 아니라, 사용자 상호작용을 위한 자주적인 **운영체제(Operating System)**로 취급합니다.

### 핵심 원칙
1.  **커맨드 중심 (Command-Centricity)**: 모든 사용자의 의도는 **커맨드 객체 (Command Object)**입니다 (예: `{ type: 'ADD_TODO', payload: { text: '우유 사기' } }`).
    - 뷰(View)는 로직을 *어떻게* 실행하는지 알지 못하며, 오직 의도를 *전달(dispatch)*하는 방법만 압니다.
2.  **순수 뷰 (Pure View)**: React 컴포넌트는 철저하게 상태의 수동적인 투영(Projection)입니다. 컴포넌트 내부에는 비즈니스 로직이 전혀 없습니다.
3.  **관할권 포커스 (Jurisdictional Focus)**: 화면은 **관할 구역(Zone)**으로 나뉩니다. 키바인딩과 상호작용은 문맥을 인식하며, 특정 구역(Zone)이 포커스되었을 때만 활성화됩니다.

## 3. 4-계층 아키텍처 (TFIZ)
이 비전을 구현하기 위해, 우리는 논리적 상태와 매핑되는 4계층 물리적 모델을 사용합니다:

- **Zone** (관할권, Jurisdiction): 공간적 영역(예: 사이드바, 메인 콘텐츠)을 정의하고 키바인딩 규칙을 강제합니다.
- **Item** (객체, Object): 상호작용 가능한 개체(예: 할 일 행, 이메일)를 나타냅니다.
- **Field** (속성, Property): 아이템 내의 특정 데이터 포인트를 나타냅니다 (예: 할 일의 텍스트).
- **Trigger** (행위, Verb): 수행할 수 있는 동작을 나타냅니다 (예: 삭제 버튼, 체크박스).

이 모델은 모든 상호작용이 구조화되고, 예측 가능하며, **커맨드 인스펙터(Command Inspector)**를 통해 완전히 관찰 가능하도록 보장합니다.
