# 워크플로우 자가 개선 변경 분석 리포트

| 항목 | 값 |
|------|---|
| 원문 | 회고로 인한 workflow 변경에 대해서 mece하게 표로 작성하고 어떤 방향성으로 진화하고 있는지 리포트를 작성해 |
| 내(AI)가 추정한 의도 | 개별 workflow 수정의 단편적 나열이 아니라, 변경 패턴에서 **구조적 진화 방향**을 읽어내고 싶다 |
| 날짜 | 2026-02-14 |
| 범위 | defineApp v5 프로젝트 회고 2회 (세션 회고 + 프로젝트 전체 회고)에서 발생한 workflow 변경 전체 |

---

## 1. 개요

defineApp v5 프로젝트 완료 후 `/retrospect`를 2회 실행하여 7개 workflow를 수정했다. 이 리포트는 변경 사항을 MECE하게 분류하고, workflow 생태계가 어떤 방향으로 진화하고 있는지를 분석한다.

---

## 2. MECE 변경 분류표

### 축 1: 변경 대상 (Workflow) × 축 2: 변경 유형 (What)

| Workflow | 단계 추가 | 단계 조건부화 | 형식 표준화 | 명령어 정정 | 연결 강화 |
|----------|:---:|:---:|:---:|:---:|:---:|
| `/project` | — | `/tdd` 스킵 조건 | 최소 루프 명시 | — | CLOSING에 `/retrospect` 호출 추가 |
| `/redteam` | "다음 단계" 섹션 | Blue Team 자동 프롬프트 | — | — | 분석→실행 브릿지 |
| `/divide` | 재분류 단계, 통합 버그 전략 | — | — | — | — |
| `/go` | 커밋 단계 | — | 멈춤 보고서 형식 (`### 완료` / `### 의사결정 필요`) | — | — |
| `/workflow` | 기존 파일 존재 확인 | 초기 질문 조건부화 | — | — | — |
| `/verify` | — | — | — | `tsc --strict` → `tsc`, `npm run build` → `npx vite build` | — |
| `/retrospect` | (전면 재작성) | — | — | — | — |

### 축 3: 변경 원인 (Why) — Problem 유형별

| Problem 유형 | 해당 변경 | 사례 |
|-------------|----------|------|
| **절차 누락** — 해야 하는데 workflow에 없음 | `/go` 커밋 단계, `/workflow` 존재 확인, `/redteam` 다음 단계 | 커밋 없이 누적 → 리스크. 기존 파일 덮어쓸 뻔 |
| **절차 경직** — 항상 실행하라 했지만 불필요한 경우 존재 | `/project` /tdd 조건부화, `/workflow` 초기 질문 조건부화 | 이미 테스트가 있는데도 /tdd 실행 요구 |
| **출구 부재** — 단계가 끝나고 다음이 없음 | `/redteam` 분석→실행 브릿지, `/project` CLOSING→/retrospect | Red Team 합의 후 "이제 뭐 하지?" 공백 |
| **형식 불일치** — workflow 명령이 실제와 다름 | `/verify` tsc/build 명령, `/go` 보고서 형식 | `--strict` 플래그가 tsconfig과 중복 |
| **판단 오류 허용** — 잘못된 분류를 수정할 절차 없음 | `/divide` 재분류 단계 | Open으로 분류한 것이 실제로는 Known (payload 키 불일치) |

### 축 4: 변경 규모 (Impact)

| 규모 | 변경 | 설명 |
|------|------|------|
| **L (구조적)** | `/retrospect` 전면 재작성 | 범용 KPT → workflow 자가 개선 루프. 역할 자체가 바뀜 |
| **L (구조적)** | `/project` CLOSING에 `/retrospect` 연결 | 프로젝트 종료 시 workflow 개선이 자동 트리거됨 |
| **M (단계 수준)** | `/redteam` Blue Team + 다음 단계 | 분석 workflow에 실행 브릿지 추가 |
| **M (단계 수준)** | `/divide` 재분류 + 통합 버그 전략 | 판단 유연성 강화 |
| **M (단계 수준)** | `/go` 커밋 + 보고서 형식 | 실행 루프의 안전장치 + 출력 표준화 |
| **S (미세 조정)** | `/verify` 명령어 정정 | 실제 프로젝트와 일치시키는 정정 |
| **S (미세 조정)** | `/workflow` 존재 확인 + 조건부 질문 | 엣지케이스 방어 |

## 3. 진화 방향 분석

### 방향 1: **경직 → 유연** (Rigid → Adaptive)

```
Before: "항상 이 단계를 실행하라"
After:  "조건이 맞으면 스킵하라"
```

- `/project`: "테스트 먼저" → "기존 테스트가 있으면 스킵"
- `/project`: 6단계 필수 루프 → "최소 `/divide` + `/verify`는 필수, 나머지 선택"
- `/workflow`: "항상 물어봐" → "이미 지정했으면 스킵"

**패턴**: 첫 번째 버전은 "최악의 경우"를 가정하고 모든 단계를 필수로 만든다. 실전에서 사용하면 "이건 불필요한데?"가 드러나고, 조건부 가드가 추가된다. **Workflow는 법률이 아니라 가이드라인으로 진화한다.**

---

### 방향 2: **단절 → 연결** (Isolated → Connected)

```
Before: 각 workflow가 독립적으로 끝남
After:  workflow가 다음 workflow를 트리거
```

- `/project` CLOSING → `/retrospect` 호출
- `/redteam` 분석 → "다음 단계" → 실행 workflow로 연결
- `/go` 실행 → 검증 통과 → 커밋 (루프 완결)

**패턴**: 개별 workflow가 "내 역할만 하고 끝"에서 "다음 단계를 안내"로 진화한다. 결국 workflow 간의 **호출 그래프**가 형성되고, workflow 생태계가 하나의 파이프라인처럼 동작한다.

```
/discussion → /project → /redteam → /divide → /verify → /go → /retrospect
                                                                    ↓
                                                            workflow 파일 수정
```

---

### 방향 3: **명시적 → 자가 교정** (Static → Self-Correcting)

```
Before: 한 번 쓰면 그대로
After:  사용할 때마다 스스로를 고침
```

- `/retrospect`의 핵심 변화: "작업을 회고" → "**workflow를 회고하고 즉시 수정**"
- `/divide`에 재분류 단계 추가: Open/Known 판단이 틀려도 수정 가능
- `/project` CLOSING에 `/retrospect` 내장: 매 프로젝트마다 자가 개선 루프 실행

**패턴**: 가장 메타적인 진화. Workflow가 **자기 자신을 개선하는 피드백 루프**를 갖게 된다. 코드의 테스트처럼, workflow에도 "회고"라는 검증 메커니즘이 내장된다.

---

### 방향 4: **추상적 → 구체적** (Abstract → Concrete)

```
Before: "보고서를 작성한다"
After:  "### 완료 | # | 작업 | 상태 | 형식으로 보고한다"
```

- `/go` 멈춤 보고서 형식 표준화
- `/verify` 명령어를 실제 프로젝트 명령과 일치

**패턴**: 추상적 지시("보고서 쓰세요")는 AI마다 다르게 해석할 수 있다. 구체적 형식을 명시하면 출력 품질이 안정화된다. **일관성은 구체성에서 온다.**

---

## 4. 진화 공식

네 방향을 하나로 압축하면:

> **Workflow는 "이상적 절차서"에서 "실전 적응형 파이프라인"으로 진화한다.**

| 세대 | 특징 | 비유 |
|------|------|------|
| **Gen 0** | 절차가 없음 | 암기 |
| **Gen 1** | 고정 절차 (모든 단계 필수) | 매뉴얼 |
| **Gen 2** ← 현재 | 조건부 절차 + 연결 + 자가 교정 | 프로토콜 |
| **Gen 3** (예측) | 맥락에 따라 동적으로 절차를 선택 | 에이전트 |

현재 workflow 생태계는 **Gen 1 → Gen 2 전환기**에 있다. `/retrospect`가 이 전환의 촉매다.

---

## 5. 해법 유형

🟢 **Known** — workflow 자가 개선은 이미 Agile/Lean의 Kaizen 원칙으로 확립된 패턴. 방향성은 자명하고, `/retrospect`가 그 메커니즘을 제공한다.

---

## 6. 인식 한계

- 이 분석은 **2회의 회고** 데이터에 기반한다. 샘플이 적어 Gen 3 예측의 신뢰도는 낮다.
- workflow 변경의 **실제 효과**는 다음 프로젝트에서 `/retrospect`를 다시 실행해야 측정 가능하다.
- "/retrospect 피로" (매번 형식적으로 회고하게 되는 현상) 가능성은 평가하지 못했다.
