## Conclusion

### Why

AI 협업에서 10일간 383개 문서가 생성되었지만, 시간이 지나면 맥락을 잃어 파편화된 노이즈가 된다. PARA는 "작업의 지도"이지 "지식의 지도"가 아니어서, 문서가 올바른 곳에 놓이지 않고, 노화를 막는 구조가 부재하다.

### Intent

문서 시스템을 **살아있는 문서(지형 기반, 덮어쓰기)**와 **죽은 문서(시간 기반, 매장)**로 이분법화하고, 개발 중 생산되는 문서는 에이전트 컨텍스트로 유지하되, 작업 완료 후 **환류(official 추출) + 매장(archive/주차)** 파이프라인을 구축한다.

### 결론 구조

```
살아있는 문서 (덮어쓰기, 지형 기반):
  official/            ← 공식 지식의 인지 지도 (소스코드 토폴로지와 동형)
    kernel/              Overview → API Reference → Glossary
    os/                  SPEC.md 기반 확장
  .agent/rules.md      ← 헌법 (강제 노출, 매 세션 읽힘)

성숙 중인 문서 (인큐베이터):
  2-area/              ← official로 아직 졸업 못 한 것 (meta, cross-cutting)

작업 중 문서 (에이전트 작업 기억):
  1-project/           ← BOARD, discussion, status (개발 중 필수)
  0-inbox/             ← 분류 전 입구

죽은 문서 (매장, 분류 없음):
  archive/2026/02/W08/ ← 연도/월/주차로 자동 매장
```

### 문서의 세 가지 실천 규칙

1. **더 잘 쓴다** = 소스코드 토폴로지에 기반한다. official/은 지형도처럼 안 변하는 구조
2. **더 잘 버린다** = 분류하지 않는다. archive/연도/월/주차에 넣고 끝
3. **덜 만든다** = 줄이지 않는다. 개발 중 에이전트 컨텍스트로 필수. **사후 환류 파이프라인이 진짜 해법**

> **한 줄 요약**: 문서를 "살아있는 것(official = 소스코드 지형 기반)"과 "죽은 것(archive/연도/월/주차)"으로 이분법화하고, 개발 중 문서는 에이전트 작업 기억으로 필수이므로 줄이지 않되, 작업 완료 시 환류(official 추출) + 매장(archive 자동 분류) 파이프라인을 구축한다.
