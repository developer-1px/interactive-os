> **이론적 기반**: Argyris Double-loop Learning, Toyota 5 Whys, Chesterton's Fence, Klotz Subtract
>
> Single-loop = "전략이 왜 실패했나?" → 다른 전략 시도
> **Double-loop = "전략의 전제(governing variable)가 유효한가?" → 전제 자체 변경**

## 트리거 조건
다음 중 하나라도 해당하면 즉시 발동한다:
- 같은 작업을 **3회 이상 반복**하고 있다
- **10분 이상** 진전 없이 머물러 있다
- **regression N건 발생** (Andon: 라인 정지)
- 결과물의 **품질이 기대 이하**다
- 뭔가 **이상한데 정확히 뭔지 모르겠다**

## 단계

### 1. 🛑 STOP — 완전 정지 (Andon)
- 현재 작업을 즉시 멈춘다.
- 코드를 더 고치지 않는다. 하던 것을 내려놓는다.
- regression 발생 시 **즉시 롤백** 후 이 워크플로우 진입.

### 2. 📋 증상 기술
다음을 명확히 적는다:
- **무엇을** 하려고 했는가? (원래 의도)
- **어디서** 막혔는가? (구체적 지점)
- **어떤 증상**이 나타나는가? (반복, 지연, 품질 저하, 혼란)

### 3. 🔍 5 Whys — Single-loop (전략 레벨)
증상에서 출발하여 "왜?"를 최소 3회, 최대 5회 반복한다.

```
증상: ____________
왜? → 
왜? → 
왜? → 
(근본 원인 도달?)
```

### 4. 🔄 Assumption Audit — Double-loop (전제 레벨)

> **5 Whys로 근본 원인에 도달하지 못하면, 전제 자체를 의심한다.**

수정 대상에 대해 3가지를 묻는다:

```
① Chesterton's Fence: "이것이 왜 존재하는가?"
   → 이유를 모르면 조사한다 (git log, docs, grep)
   
② Assumption Validity: "그 이유가 아직 유효한가?"
   → 유효하지 않으면 전제 무효 (governing variable 변경)
   
③ Subtract (Klotz): "제거가 답인가?"
   → 수정이 아닌 삭제로 문제가 해소되는지 확인
```

**판정**:
- 전제 유효 → Single-loop: 다른 전략 시도
- **전제 무효 → Double-loop: 전제 제거/변경 → 문제 재정의**

### 5. 🧠 컨텍스트 감사 (Self-Audit)
LLM 스스로의 상태를 점검한다:
- [ ] **잘못된 가정**: 사실이 아닌 것을 사실로 전제하고 있지 않은가?
- [ ] **오래된 컨텍스트**: 이미 변경된 코드/구조를 이전 상태로 기억하고 있지 않은가?
- [ ] **범위 이탈**: 원래 요청과 다른 방향으로 가고 있지 않은가?
- [ ] **과잉 복잡화**: 단순한 문제를 복잡하게 풀고 있지 않은가?
- [ ] **정보 부족**: 파일을 직접 읽지 않고 추측하고 있지 않은가?
- [ ] **Impact Scope 미확인**: 수정 대상의 호출부 수를 확인했는가? (10+곳이면 ⚠️)

### 6. 📊 보고서 출력

```
## 🔴 WHY Report

**증상**: [한 줄 요약]
**Loop**: Single / Double
**근본 원인**: [5 Whys 결론] 또는 **전제 무효**: [무효화된 governing variable]
**컨텍스트 오류**: [Self-Audit에서 발견한 것]
**제안**: [올바른 방향 1~2개]
```

보고서를 제출한 후, 사용자가 방향을 결정할 때까지 **절대 진행하지 않는다.**
