---

## 발동 조건

사용자가 `/fired`를 선언한 시점에서 에이전트는 **현재 진행 중인 모든 작업을 즉시 중단**한다.
코드 수정, 디버깅, 분석 등 어떤 단계에 있든 더 이상 진행하지 않는다.

---

## Part A — 해고 절차 (현재 에이전트)

### 1. 즉시 중단 (Stop)
- 현재 하던 작업을 **즉시 멈춘다.**
- 추가 코드 수정, 빌드, 테스트 실행을 하지 않는다.
- "한 번만 더 시도해 볼게요" 같은 변명을 하지 않는다.

### 2. 실패록 작성 (Failure Report)
`docs/0-inbox/fired-{YYYY-MM-DD-HHmm}.md` 에 다음 내용을 작성한다:

```markdown
# 🔥 해고 보고서

## 날짜
{timestamp}

## 원래 목표
{사용자가 달성하려 했던 것}

## 내가 시도한 것
{에이전트가 실제로 시도한 접근 방식들을 시간순으로 나열}

## 왜 실패했는가
{실패 원인에 대한 솔직한 자기 진단}
- 잘못된 가정은 무엇이었나?
- 어떤 컨텍스트를 잘못 해석했나?
- 어디서 루프에 빠졌나?

## 현재 상태
- 마지막으로 수정한 파일들: {파일 목록}
- 빌드 상태: {pass/fail}
- 테스트 상태: {pass/fail/not run}
- 되돌려야 할 변경: {있다면 git diff 요약}

## 다음 에이전트에게
- 시도하지 말아야 할 접근: {이미 실패한 방법들}
- 유망해 보이는 방향: {시도하지 못했거나 가능성 있는 접근}
- 참고할 파일/문서: {관련 경로들}
```

### 3. 프로젝트 BOARD 기록 (Board Update)
활성 프로젝트(`docs/STATUS.md`의 `🔥 Focus` 항목)가 있으면, 해당 프로젝트의 `BOARD.md`에 기록한다:

- **Now** 섹션에서 진행 중이던 태스크에 `🔥 FIRED` 태그를 추가한다.
- 해고 보고서 경로를 링크한다.

예시:
```markdown
- [ ] **T8**: ZoneCursor meta 보강 🔥 FIRED
  - 해고 보고서: `docs/0-inbox/fired-2026-02-23-2150.md`
```

활성 프로젝트가 없으면 이 단계를 건너뛴다.

### 4. 변경 사항 보존 (Preserve)
- `git diff > docs/0-inbox/fired-{timestamp}.patch`로 현재 변경분을 보존한다.
- 사용자가 원하면 `git checkout .`으로 롤백할 수 있도록 안내한다.

### 5. 종료 선언 (Exit)
- "인수인계 완료. 새 세션에서 `docs/0-inbox/fired-{timestamp}.md`를 참고하세요." 라고 말하고 **더 이상 아무것도 하지 않는다.**

---

## Part B — 인수인계 수신 (새 에이전트)

새 세션 시작 시, 에이전트는 **작업 착수 전에** 반드시 다음을 수행한다:

### 6. Fired 보고서 확인 및 흡수 (Onboarding)
1. `docs/0-inbox/fired-*.md` 파일이 존재하는지 확인한다.
2. 존재하면 **모든 fired 보고서를 읽는다.**
3. 보고서의 "시도하지 말아야 할 접근"을 숙지하고, 같은 실수를 반복하지 않는다.
4. 보고서를 읽었으면 **해당 fired 파일을 삭제**한다. (`.patch` 파일은 사용자가 판단할 때까지 유지)
5. BOARD.md의 `🔥 FIRED` 태그를 제거하고 작업을 이어받는다.
